## Reddit Post — r/LocalLLaMA

**Title:** Built a week-over-week learning stats dashboard for a local AI tutor — no telemetry, all local SQLite, 571 lines — and the empty-state problem I haven't fully solved

**Body:**

OpenTutor is an open-source AI learning platform that runs entirely on your own hardware. I shipped a dashboard overhaul this week — weekly stats with deltas, cross-course mastery tracking, a composable summary workspace block, and a goal timeline view. I want to write about the technical decisions and the tradeoff I made around empty states.

**What shipped**

The dashboard now has two new sections. `WeeklyStatsSection` renders three numbers — study time in minutes, quiz accuracy percentage, and active days — alongside the same metrics from the prior ISO week. The signed deltas use green/red coloring and fall back to null if there's no prior week to compare against. The data comes from a new `getWeeklyReport()` client call against `/api/progress/weekly-report`, which buckets `StudySession` records from SQLite by ISO week number. All local, no external calls.

`MasteryOverviewSection` pulls all active courses, reads their LOOM mastery scores from `LearningOverview`, and renders them as a sorted list with text-based mastery percentages. The mastery data was already being computed by the existing knowledge graph system — this component just surfaces it in a cross-course view.

The third piece is a new workspace block: `summary-block.tsx` (118 lines), registered as block type `"summary"` in the block registry. It fetches `getCourseProgress()` and `getReviewSession()` in parallel and renders a 2×2 grid of stat cards: mastery %, nodes mastered of total, FSRS review items due, total study time formatted as hours/minutes. It's now part of the default block layout for `self_paced` and `exam_prep` modes.

The fourth piece is a `TimelineView` component (241 lines) in the plan section: a new tab that reads study goals, sorts them by deadline, and renders urgency-coded labels — overdue (destructive), soon (7 days, warning), normal (30 days, brand), future (muted). Goals without dates fall to the bottom.

**The local-first problem with analytics**

The honest tension: every analytics surface I build for OpenTutor has the same cold-start problem. The weekly stats comparison requires data from a previous week. The mastery overview requires at least one course with tracked progress. The summary block shows empty cards if no session has completed yet.

I made a deliberate choice: ship the surfaces anyway and let them populate with use rather than gate them behind a data threshold. The alternative — only showing these sections once sufficient data exists — felt worse. Showing nothing teaches the user nothing about what's coming. Showing an empty state at least establishes that the surface exists and what it will show.

The gap I haven't closed: there's no "here's what you'll see after your first week" preview or skeleton. The empty state is just empty. I may add placeholder text with expected values ("e.g., ~45 min/week after a few sessions") as a way to set expectations without fabricating data.

**Architecture note**

The block registry is the part of this I'm most happy with. Adding the summary block required 8 lines of change to `registry.ts` and `templates.ts` — the registration, the default assignment to two layout templates, and the i18n strings. The block itself is self-contained: it fetches its own data, manages its own loading/error states, and renders within whatever grid slot it's assigned. Adding or removing it from a layout is a one-line change to a template definition. That composability was a deliberate design goal from early on, and it held up here.

Tech stack: FastAPI, Next.js, SQLite, FSRS 4.5. Runs locally on Ollama or any OpenAI-compatible API.

If you've built local-first analytics — no centralized data, one user per deployment — how do you handle the gap between "data surfaces exist" and "data doesn't exist yet"? Synthetic previews? Gated rendering? Something else?

## Reddit Post — r/Anki

**Title:** My local Anki-like tutor now shows a daily digest block inside the study workspace — mastery %, review due, total time, all from local data

**Body:**

I've been building OpenTutor — an open-source, local-first study tool that uses FSRS 4.5 (same scheduling algorithm as newer Anki versions) — and I shipped a meaningful dashboard improvement this week. I want to share what it shows and why I think it's useful for spaced repetition workflows specifically.

**The new summary block**

The workspace now has a "summary" block — a composable panel that sits alongside the flashcard and quiz blocks while you're studying a course. It shows four numbers:

- **Average mastery** — the mean LOOM knowledge score across all concepts in the course. A concept's mastery tracks correctness over time, not just last-attempt outcome.
- **Mastered count** — how many concepts have crossed the mastery threshold, out of total.
- **FSRS review due** — how many items are scheduled for review today (pulled from the same review session logic that drives the flashcard block).
- **Total study time** — cumulative minutes logged across all sessions for this course.

The intent: before you start a session, you glance at the summary block and have an immediate answer to "where am I?" If review due is 47, today is mostly review. If it's 3 and mastery is 34%, today is probably new material and diagnostic work.

**The dashboard additions**

At the top level, the dashboard now shows a WeeklyStats section with this week vs. last week deltas for study time, quiz accuracy, and active days. For those of you who track streaks in Anki: this is the closest OpenTutor has to a streak-adjacent metric. Active days shows how many days you had at least one session this week vs. last week.

There's also a MasteryOverview section that shows all your active courses ranked by mastery score. If you run multiple subjects like I do — one course for a technical topic, one for language study — it lets you see at a glance whether one is lagging. Anki users who maintain multiple decks know this problem: it's easy to unconsciously neglect a deck that isn't giving you due reviews right now, even though your long-term retention on it is dropping.

**The FSRS connection**

The review due count in the summary block comes from the same FSRS scheduling pass that fills the flashcard block. OpenTutor uses FSRS 4.5 with per-card stability tracking. When the summary block shows "12 reviews due," those are the same cards you'd see in the flashcard block if you opened it. It's not a separate calculation.

The mastery score is separate from FSRS stability — it tracks concept-level correctness aggregated across all question types, not just flashcard recall. You can have a high FSRS stability on a flashcard (you're reliably recalling it) and still have low mastery on the underlying concept (you keep getting the quiz questions wrong on application-level questions). The summary block shows mastery, not stability, because mastery is harder to game with passive recognition.

**What I haven't figured out**

The "mastered" threshold is currently set at mastery >= 0.8. That might be too low — or too high — depending on what you're studying and what "mastered" means to you. I haven't made it configurable yet. I'm not sure it's useful to expose a number that most users won't know what to do with.

For Anki users who mix OpenTutor with Anki: do you find that mastery percentage or retention rate is more useful as a course-level summary metric? I'm curious what signals actually change your study behavior vs. which ones you check once and ignore.

## IndieHackers Post

**Title:** Week 25: The dashboard tracked everything. It just didn't tell you anything. Here's what I changed and the empty-state tradeoff I'm not sure about.

**Body:**

OpenTutor has had a detailed internal model of your learning state since early on. LOOM tracks concept mastery scores. FSRS tracks card intervals and stability. The activity engine logs session timestamps, study minutes, quiz accuracy per session, error streaks. All of it was computed, stored, and used to drive adaptive behavior in the tutor.

What the dashboard showed: course titles. A "continue" button. When you last opened each course.

That mismatch bothered me for a while. I kept adding capabilities to the adaptive backend and not connecting them to anything you could see. The dashboard was a navigation surface, not a feedback surface. You couldn't tell if studying was working.

This week I shipped three things to close that gap.

**Weekly stats with week-over-week deltas.** The dashboard now shows study time, quiz accuracy, and active days for the current ISO week, alongside the prior week's values and a signed delta. It's three numbers — deliberately small. I didn't want to build a chart. Charts in personal productivity tools often tell you less than you think they do: they look informative and then you stop reading them after a week. Three numbers you actually compare week to week is more useful than a sparkline you stop noticing.

**Cross-course mastery overview.** If you have multiple courses, the dashboard now shows all of them ranked by mastery score, with the average mastery displayed as a percentage. The mastery scores were already being computed — the knowledge graph tracks them per concept, and the learning overview endpoint aggregates them. I just added the surface. Took 60 lines of React.

**A summary block inside the workspace.** This is the piece I think matters most for day-to-day use. It's a composable workspace block — the 13th in the registry — that shows a daily digest: mastery %, mastered concept count, FSRS reviews due today, cumulative study time. It lives in the workspace alongside the flashcard block and the quiz block. You open a course and the summary block answers "where am I right now?" without you having to navigate anywhere.

The part I keep second-guessing: all three of these surfaces are empty for new users. The weekly stats comparison requires data from a prior week. The mastery overview requires at least one course with tracked mastery. The summary block shows zeroes until you've had a few sessions. I shipped empty-state handling — components return null or render placeholder text when there's no data — but "graceful empty" is still empty.

I made the call to ship them anyway. The alternative — gating these sections behind a data threshold — felt like hiding the product from the people who most need to see what it does. But I'm aware that a new user's first impression of the dashboard is now a set of blank widgets. I haven't fixed that.

The broader challenge with a local-first tool: I don't have aggregate usage data. I don't know which of these surfaces users actually look at. I'm building analytics for a system where I can't measure whether the analytics are useful. I check my own study data, I think about what I'd want to see, and I ship. That's the only loop available to me right now.

OpenTutor is fully open-source — the dashboard code is all in `apps/web/src/app/_components/dashboard-sections.tsx`. The summary block is `apps/web/src/components/blocks/blocks/summary-block.tsx`.
