## Reddit Post — r/LocalLLaMA

**Title:** Built a cognitive load auto-tuner for a local LLM tutor: 12 signals, 20-message sliding window, no ML — and the design tradeoff I'm uncertain about

**Body:**

OpenTutor is an open-source learning platform that runs entirely on your own hardware — Ollama, LM Studio, or any OpenAI-compatible endpoint. It tracks cognitive load in real time using 12 behavioral signals and injects a guidance fragment into the tutor's system prompt to adapt explanation depth. This week I want to write about a bug in that system that took me longer than it should have to notice, and the deliberately simple fix.

**The problem: signal saturation**

The 12 signals include things like: recent error rate, message brevity, answer hesitation (time from question display to submission), NLP frustration detection via a lightweight LLM call every 3 messages, wrong-answer streaks, message gap (pause between messages), repeated unmastered errors, and quiz performance trends. Each signal is a float in [0.0, 1.0], multiplied by a weight, summed, and clamped to get a final load score in [0.0, 1.0].

The problem: some learners have a high unmastered error count as their baseline. Not because they're in crisis — they might be steadily working through a hard course. But their `unmastered_errors` signal stays near 1.0 every session. So their cognitive load score is permanently near the "high" threshold, the tutor perpetually fires high-load guidance, and the adaptive behavior stops being adaptive. It's fighting a signal that just describes who this learner is, not how they feel right now.

**The fix: a sliding window dampener**

I added a per-user weight auto-tuner (`cognitive_load_tuning.py`, 136 lines). After each load computation, it records the signal values in a per-signal deque of length 20. If the running mean of any signal exceeds 0.65, the weight multiplier for that signal decays by 10% per update, down to a floor of 0.40. When the signal drops below threshold, the multiplier recovers at 4% per update, up to a ceiling of 1.0.

Constants used: `WINDOW_SIZE=20`, `HIGH_SIGNAL_THRESHOLD=0.65`, `WEIGHT_DECAY_FACTOR=0.90`, `MIN_WEIGHT_RATIO=0.40`, `RECOVERY_FACTOR=1.04`. These are deliberately conservative — the tuner is meant to adapt to a learner's baseline range, not suppress signals entirely. A learner with genuinely elevated errors still contributes that signal at 40% weight minimum.

The whole thing is in-memory with a bounded LRU-style cache (200 users max), evicting the least active user when full. No database writes, no persistence between restarts. Phase 2 would add persistence if it turns out these multipliers matter enough to preserve across sessions.

**Why not ML?**

Two reasons. First, I don't have enough data. This is a local-first tool where each deployment has one user; there's no population-level training set. Second, the sliding window approach is interpretable in ways that matter for a learning tool: if I see the tuner dampening a signal, I can explain exactly why. A neural load estimator that outputs a number without a per-signal breakdown is harder to debug and harder to trust.

The comment in the code says it best: "This is intentionally simple: no ML, just a sliding window mean + a small dampening step. Phase 2 can add Bayesian updating if needed."

**The floor constant I'm uncertain about**

I set `MIN_WEIGHT_RATIO=0.40` — no signal can be suppressed below 40% of its configured weight. My reasoning: distinguish between "this student runs high on errors as a baseline" (dampen) vs. "this student is actually in crisis" (don't suppress entirely). But 0.40 is a guess calibrated on my own usage patterns. If anyone has built adaptive systems with similar signal-dampening logic, I'd like to know how you chose your floors.

**What shipped alongside it**

The same commit added a locked overlay to block cards in the workspace. If a block's unlock conditions aren't met — based on learning progress, active mode, or time-in-course — it renders with a blurred overlay, a lock icon, and a hint for how to unlock it. Previously the block would just show empty content with no explanation. Small UX change, but it prevents the "why is this empty?" confusion.

OpenTutor tech stack: FastAPI, Next.js, SQLite, FSRS 4.5. Everything runs locally.

Have you built systems that need to separate persistent baseline signals from transient spikes? What approach did you use, and what went wrong first?

## Reddit Post — r/Anki

**Title:** My local AI tutor was always telling me I was overloaded. Turned out the detector was wrong. Here's what I changed.

**Body:**

I've been building OpenTutor — an open-source, local-first study tool that runs on Ollama — and one of its core features is real-time cognitive load detection. The idea: measure behavioral signals while you're studying and adjust how the AI tutor explains things based on whether you seem overloaded. Under high load, the tutor gives shorter explanations, uses more examples, simplifies questions. Under low load, it can go deeper.

The system tracks 12 signals: error rate, session length, message brevity, help-seeking language, quiz performance trends, answer hesitation (how long before you submit), wrong-answer streaks, NLP frustration/confusion detection, message gap (pause before responding), repeated unmastered errors, and a few others. Each signal is weighted, summed, and normalized to a 0–1 score.

The problem I noticed after using it for a while: if you have a lot of cards with unmastered errors — maybe you're doing a big review push through a hard course — the system reads you as high-load permanently. Every session, every message. The adaptive guidance fires constantly and eventually becomes background noise you stop noticing.

The fix I shipped: a sliding window over the last 20 load computations per signal. If a signal stays above 65% for a while, its weight decays by 10% per step, down to a floor of 40%. The reasoning: if your error backlog has been high for 20 sessions straight, that's your baseline — it's not telling me anything new about right now. When the signal drops back, the weight recovers gradually.

Why this matters for spaced repetition specifically: the cognitive load score feeds into FSRS card ordering. Under high load, easier cards (higher stability = easier recall) get surfaced first. That's borrowed from cognitive load theory — when working memory is constrained, starting with material you partially know frees capacity for the harder stuff. But if the load score is saturated and wrong, the reordering fires when it shouldn't. Fixing the saturation problem made the reordering actually useful.

What I haven't figured out: the right floor value. I set 40% — no signal can be dampened more than 60%. That might be too conservative or not conservative enough. The only way to know is more usage data, which is slow to accumulate with a local-first tool.

The other thing that shipped in the same commit: blocks in the learning workspace now show a lock icon and an explanation when they're not yet unlocked, instead of silently showing empty content. Flashcards, knowledge graph, spaced review — they all unlock progressively as you engage with the material. The visual feedback makes the progression feel intentional instead of broken.

If you use Anki alongside any adaptive or AI tools — how do you handle the problem of "this signal is always high for me personally, so it's stopped being informative"?

## IndieHackers Post

**Title:** Week 24: I built a cognitive load detector. Then I noticed it was always maxed out for some learners. That's a different kind of broken.

**Body:**

OpenTutor tracks 12 behavioral signals to estimate whether a learner is overloaded in real time. Error rate, message brevity, answer hesitation, frustration detected in text, wrong-answer streaks, pauses between messages. The score runs 0–1, and the AI tutor adjusts its behavior based on it — shorter explanations, more examples, simpler questions when load is high.

I was pretty happy with this system. Then I used it heavily myself for a few weeks and noticed something: if you have a lot of unmastered errors piled up — which happens when you're grinding through a hard topic — the load score reads near 1.0 every single session. Not because you're in crisis. Just because your baseline error count is high.

The guidance fires constantly. The tutor keeps offering to slow down and simplify. And at some point you just stop noticing. The adaptation became wallpaper.

The problem has a name in signal processing: saturation. When a signal is always near its ceiling, it stops carrying information about the current moment. It just describes you.

I fixed it with a sliding window. After each load computation, the system records each signal's value in a rolling buffer of length 20. If a signal's average over that window exceeds 0.65, its weight multiplier decays by 10% — each subsequent update, 10% smaller — down to a floor of 40%. When the signal drops below threshold, it recovers at 4% per update.

No machine learning. No database writes. A deque and some arithmetic. 136 lines.

The part I'm uncertain about: the 40% floor. The reasoning is: I want the tuner to adapt to someone's baseline, not erase it. A learner who genuinely has a high error backlog should still carry some load signal from that fact — just not full weight. But 40% is a number I chose from first principles and a few hours of self-testing. It might be wrong. I won't know until more people use it.

The same commit shipped something more visible: a locked overlay on workspace blocks. OpenTutor uses composable "blocks" — flashcards, quiz, knowledge graph, timeline, progress, notes — that unlock progressively based on your engagement with a course. Previously, locked blocks showed empty content with no explanation. You'd open a course, see a blank flashcard block, and wonder if something was broken. Now there's a lock icon, a label, and a short hint for how to unlock it. One of those UX fixes that's obvious in retrospect.

I also added a Tier-2 ambient agent action: when the monitor detects you've been studying for a while but have no AI-generated notes or practice questions yet, it proposes generating them — but doesn't do it automatically. You get a pending card in the workspace that you can approve or dismiss. The distinction between "do it for me" and "ask first" was deliberate. Content generation changes your course. I don't want a background process making that call without you.

The honest challenge with OpenTutor right now: I'm building adaptive behavior for a system that needs real usage data to validate, and "local-first, runs on your machine" means I don't get telemetry. I'm tuning constants against my own study patterns and hoping they generalize. They might not.

OpenTutor is fully open-source. If you're using it and the cognitive load system is misbehaving for you — firing constantly, or never firing — I'd genuinely like to know. The constants are tunable in config, and I want to get them right.

What's your threshold for shipping an adaptive feature you can't fully validate yet?
