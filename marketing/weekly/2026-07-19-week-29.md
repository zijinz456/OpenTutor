## Reddit Post — r/LocalLLaMA

**Title:** I added LLM-graded coding questions to my local AI tutor — no execution sandbox, just the model as judge

**Body:**

OpenTutor is an open-source, local-first AI learning platform — FastAPI + Next.js + SQLite, runs on Ollama or any OpenAI-compatible endpoint. The core feature is quiz generation from your study materials. In April I added a coding question type and had to make a grading decision that will annoy some people.

**The decision**

I didn't build a code execution sandbox. The grader sends the question, the reference answer, and the student's code to the same LLM running the session and asks: does this code correctly solve the problem?

The prompt explicitly tells the model to accept any functionally equivalent implementation — single vs. double quotes don't matter, spacing doesn't matter, the logic does. It asks for a JSON response: `{"is_correct": true/false, "feedback": "brief reason"}`. The feedback is capped at 30 words. Total implementation: 63 lines.

**Why no sandbox**

OpenTutor runs locally. Not everyone has Docker. Not everyone wants to expose arbitrary code execution on their machine, even in a container. A managed execution service (Judge0, Piston, etc.) is an external dependency that defeats the local-first premise. I wasn't willing to add one as a requirement.

The question "does this code solve the problem?" is genuinely something a capable LLM can answer for most learning contexts. A student writing `def factorial(n): return 1 if n <= 1 else n * factorial(n - 1)` versus the reference `def factorial(n): if n <= 1: return 1; return n * factorial(n-1)` — the model handles that correctly.

**Where it breaks down**

Edge cases the LLM doesn't catch. Runtime behaviors it can't reason about (memory, concurrency, language-specific UB). Code that's structurally similar to the reference but wrong at the boundary. These are real failure modes and I'm not pretending they aren't.

The grader is marked "best-effort" in the codebase. If the LLM call fails or the parse fails, it returns `{"is_correct": False, "feedback": "Grading unavailable."}` — incorrectly failing the student, but immediately, without a timeout. Wrong answers feed the same downstream pipeline as all other question types: error classification, LOOM concept mastery update, automatic diagnostic problem derivation, prerequisite gap check.

**The question generation side**

The quiz extraction prompt now includes `coding` as one of 8 question types. The model is told to generate coding questions only for programming or technical content. The example in the prompt is a Python factorial recursion. In practice, the model picks the question type correctly based on what it's reading — you won't get a coding question from a history chapter.

Each coding question is annotated with `bloom_level: "apply"` and `skill_focus: "implementation"` by default. In the Bloom taxonomy framing, coding questions represent the "apply" level — the student isn't recalling or recognizing, they're producing. The quiz generator uses difficulty_layer 2 (standard application) for most coding questions.

**What I'd add with more time**

A fallback to deterministic grading for specific question patterns (write a function that takes X and returns Y — compare against actual outputs for 3-5 test cases). This is tractable without a full sandbox if you constrain the question format. It would handle the edge-case failures the LLM misses. I haven't built it because the LLM grader works well enough for the question types that actually get generated.

OpenTutor repo: https://github.com/zijinz456/opentutor

## Reddit Post — r/Anki

**Title:** I added coding questions to my FSRS-based AI tutor — the Bloom taxonomy level that Anki never reaches

**Body:**

I maintain OpenTutor, an open-source AI learning platform built around FSRS 4.5 (same algorithm as Anki since 23.10). In April I added a coding question type to the quiz generator. I want to explain why this matters in learning science terms, because I think it's the most interesting gap between Anki and what an AI tutor can do.

**The Bloom taxonomy gap**

Anki is a recall machine. Its entire mechanism is optimizing one thing: does the learner recognize or reproduce this fact at the right time? That's Bloom levels 1 (remember) and 2 (understand). FSRS schedules reviews beautifully at those levels.

Multiple choice and true/false — the other common quiz formats — don't do much better. They test recognition, not production. Even short-answer questions (which OpenTutor also supports) mostly test whether you can retrieve and write the thing.

Coding questions test at Bloom level 3: apply. The student can't recognize or recall their way to a working implementation. They have to produce something. That production failure is different in kind from a recall failure, and it points to a different gap in the knowledge graph.

**How OpenTutor handles coding questions**

The quiz generator extracts or generates coding questions from your study materials for programming or technical content. Each question has a reference answer and explanation. The student writes code in a textarea and submits.

Grading is LLM-based — no code execution, no sandbox. The model evaluates functional equivalence, not literal string matching. A student who writes a correct but differently-formatted solution gets credit.

Wrong coding answers feed the same pipeline as all other wrong answers:
- Error classification (conceptual gap, procedural error, etc.)
- LOOM knowledge graph mastery update for the relevant concepts  
- Automatic diagnostic problem derivation (a simpler version of the same concept)
- Prerequisite gap check against the knowledge graph

The FSRS scheduling layer sees coding results the same way it sees any other result — correct or incorrect — and adjusts intervals accordingly. The two systems don't interact at the algorithm level; they're decoupled by design.

**What this means for learning**

If you've used Anki to study programming, you know the gap: you can have a card for "what does the `yield` keyword do in Python" at a 30-day interval (well-retained) and still not be able to write a generator correctly when you need one. That's the Bloom gap. The card tests level 1 or 2. The actual skill is at level 3.

Coding questions in OpenTutor don't replace spaced repetition for recall. They sit alongside it, at a different level of the taxonomy. The intent is that when OpenTutor detects a pattern of wrong answers on coding questions for a concept, it's a different diagnostic signal than wrong answers on recall questions for the same concept.

**The honest caveat**

The LLM grader has failure modes — edge cases it misses, runtime behaviors it can't reason about. It's not a substitute for actually running code. For the level of accuracy needed in a learning context (did the student get the idea right?), it works. For production code review, it wouldn't.

OpenTutor runs fully locally — Ollama by default. Repo: https://github.com/zijinz456/opentutor

Anki gets you to Bloom level 2 reliably. What tools have you found that actually close the gap to level 3 for technical subjects?

## IndieHackers Post

**Title:** I added coding questions to my AI tutor and had to decide: execution sandbox or LLM judge?

**Body:**

OpenTutor is an open-source, local-first AI learning platform I've been building. It generates practice questions from your study materials and uses FSRS 4.5 for spaced repetition. In April I shipped a feature I'd been thinking about for a while: coding questions.

And then I had to make a grading decision that I'm still not sure I got right.

**The obvious path I didn't take**

Code execution sandbox. You submit your code, it runs, we compare the output. This is how competitive programming judges work (LeetCode, HackerRank, etc.). It's the "correct" answer.

But OpenTutor runs on your local machine. Some people run it on a laptop with Ollama. Some on a home server. Adding a Docker requirement or an external execution API defeats the local-first premise. I wasn't going to do it.

**What I shipped instead**

63 lines. The LLM sees the question, the reference answer, and your code. It's told: accept any functionally equivalent solution. Don't penalize style. Look at the logic. Return `{"is_correct": bool, "feedback": "under 30 words"}`.

That's the whole grader. No AST comparison, no output testing, no test cases. Just the model looking at two pieces of code and deciding whether they solve the same problem.

**Whether it works**

Mostly. For the kinds of questions the quiz generator actually produces — implement this algorithm, write a function that does X — the LLM is good at recognizing equivalent implementations. A student who writes a working but stylistically different factorial function gets credit.

Where it struggles: runtime edge cases the model can't simulate. Code that looks structurally correct but fails at a specific input. Language-specific behavior the model's training doesn't cover well. These are real failure modes. The grader is labeled "best-effort" in the code.

When it fails, the system returns `is_correct=False`. The student gets told they were wrong when they might have been right. That's a bad outcome. I chose it over the alternative (spinning on a timeout, or requiring infrastructure they might not have) because at least it's immediate and consistent.

**What surprised me**

I expected to see a lot of false negatives — students with correct code getting marked wrong. In the sessions I've reviewed, that's not the dominant failure mode. The more common problem is the LLM being generous: accepting code that's close but has a real logical error, especially around edge cases. That's actually better for a learning context than a harsh false negative, but it means I can't rely on the grader to catch subtle bugs. It grades intent better than it grades correctness.

**The downstream machine**

Wrong coding answers feed the same pipeline as every other wrong answer in OpenTutor: error classification, LOOM knowledge graph mastery update, automatic diagnostic problem generation, prerequisite gap check. The coding question type isn't special in terms of what happens after you get it wrong — it's just different input to the same learning science machinery.

**What I'd add if I were doing it again**

A lightweight test-case mode for the subset of questions that have verifiable outputs. Something like: define 3 input/output pairs, run them deterministically. Not a full sandbox — just `subprocess.run` with a timeout, in an environment I control, for constrained inputs. That would handle the edge-case failures without adding major infrastructure.

I haven't built it because the LLM grader handles the questions that actually get generated well enough. If the question type gets used more and I see more grading failures, that's the next step.

OpenTutor: https://github.com/zijinz456/opentutor

If you've made a similar tradeoff — "good enough now" vs. "right but harder" — how did it hold up over time?
