## Reddit Post — r/LocalLLaMA

**Title:** I deleted 4,500 lines from my local AI tutor in one commit — the dead code audit that clarified what the project actually is

**Body:**

OpenTutor is an open-source, local-first AI learning platform — FastAPI backend, Next.js frontend, SQLite, runs on Ollama or any OpenAI-compatible endpoint. In March I deleted 4,500 lines of it in a single commit. I want to write honestly about what was in those 4,500 lines, why they existed, and what the deletion actually cost.

**What got cut**

The most interesting deletion was `graph_memory.py` (151 lines) — an LLM-driven entity and relationship extraction pipeline that pulled structured knowledge out of tutor conversations. The design was borrowed directly from mem0's architecture: extract named entities, extract relationships between them, store them in a graph, use similarity search for retrieval. It was a real implementation, not a stub.

The problem: OpenTutor already has LOOM, a FSRS-backed knowledge graph that tracks concept mastery. The two systems were never connected. Graph memory was building a parallel representation nobody queried. I deleted it because LOOM is the actual thing, and graph_memory.py was aspirational infrastructure for a use case that hadn't materialized.

`session_export.py` (239 lines) was a SQLite export service — it would package all of a user's learning data (memories, preferences, FSRS progress, chat history) into a portable database for backup or transfer. This is the right local-first feature to eventually have. It had no frontend endpoint. It had no user-facing trigger. It had complete test coverage and no callers.

The workspace router (72 lines) provided REST endpoints to browse, download, and delete files created by the tutor agent in a sandboxed directory. I wrote it during a period when I thought the agent would produce a lot of downloadable artifacts. The agent's behavior changed. The router stayed.

**Why this took so long to do**

These modules felt like assets, not liabilities. They were well-structured. They had docstrings. Some had tests. Deleting them felt like discarding work rather than cleaning up a mess.

What eventually pushed me to do it: I was onboarding a potential contributor and found myself explaining what graph_memory.py did, why it wasn't connected to LOOM, and why session_export.py had no UI caller. These were reasonable questions with uncomfortable answers. "That exists in case we need it later" is not an architecture. It's a to-do list disguised as a module.

**What it actually cost**

One day of work. Tracing import chains, confirming nothing live called the deleted modules, cleaning up test files that referenced them. One CI green. 79 files touched, 4,500 lines down, everything still passing.

The codebase is measurably faster to navigate now. More importantly, it's more honest: what you see in the repo is what the system actually does.

**What survived**

FSRS 4.5 scheduling. LOOM knowledge graph. LECTOR semantic review. The 12 composable learning blocks. The tutor agent with cognitive load signals and adaptive mode switching. These are the things OpenTutor is actually about. Deleting everything else made that clearer.

If you run local models and have thoughts on graph-memory architectures (or when to actually build one vs. when it's premature), I'd be interested in the discussion. OpenTutor's GitHub: https://github.com/zijinz456/opentutor

What's the largest dead-code deletion you've done on a project you still actively maintain?

## Reddit Post — r/Anki

**Title:** Deleted 4,500 lines from my open-source spaced repetition tutor — what survived tells you what the project is actually about

**Body:**

I maintain OpenTutor, an open-source AI learning platform built around FSRS 4.5 (same algorithm Anki uses since 23.10). A few months ago I deleted 4,500 lines of it in one commit — roughly 15% of the codebase. I want to write about what got cut, because I think what survived says something useful about what actually matters in a learning tool.

**What got cut**

Graph memory (151 lines): an LLM-based entity extraction pipeline that tried to build a parallel knowledge graph from tutor conversations. Deleted because LOOM, the FSRS-backed knowledge graph that tracks concept mastery, was already doing the real work.

Session export (239 lines): a portable SQLite export of all learning data for backup. Right idea, no frontend caller, nobody asking for it yet.

Nine shadcn/ui components: avatar, card, dropdown-menu, resizable, separator, sheet, tabs, textarea, command. All imported, none rendered in the actual interface.

A service worker (sw.js, 135 lines) from an old offline-first experiment. Offline mode was deprioritized when I realized local-first already handles the core concern differently.

**What didn't get cut**

FSRS 4.5 scheduling. This is the heart of OpenTutor's retention system and it was never a deletion candidate. Every card, every quiz attempt, every review interval flows through FSRS.

LOOM: the knowledge graph that maps your mastery of individual concepts within a course. When the tutor decides whether to introduce new material or review old, it's querying LOOM.

LECTOR: the semantic review system that checks whether your written explanations of a concept actually match the source material, not just whether you wrote something.

The 12 learning blocks and the adaptive unlock system. The cognitive load signals.

**The parallel to spaced repetition itself**

One reason I let dead code accumulate: it felt expensive to delete things I'd built. Same loss-aversion that makes people hold onto flashcard decks they no longer review — the cards feel like sunk cost, not dead weight.

Spaced repetition works partly because it forces you to let things go. If you don't review a card and it lapses, FSRS lets the interval decay. The card doesn't disappear, but you stop scheduling it. Dead code has no equivalent mechanism. You have to do it manually.

After the deletion: the codebase is smaller, tests run faster, and the things that matter are easier to find.

OpenTutor runs entirely on your own hardware — Ollama by default, or any OpenAI-compatible endpoint. Full repo: https://github.com/zijinz456/opentutor

## IndieHackers Post

**Title:** I deleted 4,500 lines of my own product in one commit. Here's what that felt like.

**Body:**

OpenTutor is an open-source, local-first AI tutor I've been building for about eight months. In March I deleted 4,500 lines of it in a single commit. I want to write honestly about the experience — not the technical part, but the part before the technical part.

**How I noticed**

I was trying to add a feature and kept running into modules I didn't fully recognize. I'd written them, but months ago, in a different context, with a different understanding of what the project was going to be. I'd open a file, read the docstring, and think: "Right. We were going to do that."

The thing about dead code in an early-stage project is that it doesn't look like dead code. It looks like ambition. `graph_memory.py` had a detailed docstring explaining entity extraction and similarity-based node merging. `session_export.py` had complete test coverage. These weren't abandoned files. They were finished modules for features that the product had quietly pivoted away from.

**The audit**

I spent an afternoon asking a simple question for every module in the codebase: is anything calling this? And if yes: is the thing calling it also being called?

The workspace router had no active callers. The session export service had tests but no API endpoint. Graph memory had been decoupled from LOOM six weeks earlier, when I decided LOOM was the canonical knowledge graph, and nobody had cleaned up afterward.

Nine UI components from the shadcn/ui library were imported into the registry but not rendered anywhere in the actual interface. A service worker from an offline-mode experiment. A system analytics router with 196 lines of endpoint code that had been disconnected from the frontend.

**The fear**

Every one of these was a "we might need it later" case. And I'm honest enough to say: that reasoning is usually correct. You often do need things later. I've deleted code and re-written it before.

But the cost of keeping it was real and compounding. Every time I navigated the codebase, the dead modules shaped my mental model. When I thought about what OpenTutor does, I'd include things it actually doesn't do. When I estimated how long something would take to build, I'd account for systems that weren't connected.

I deleted all of it. 4,500 lines, 79 files, one commit, March 31.

**What it cost**

One day of work. Tracing imports, confirming nothing live relied on the deleted code, cleaning up tests that referenced removed services. Nothing broke. CI stayed green.

**What it felt like after**

The codebase is smaller. That's the literal thing. But the more useful thing: it's more honest. What you see in the repo is what the product does. I can explain the system in one paragraph without caveats.

I still think session export is the right feature to build — eventually. A local-first tool should let you take your data with you. But building it when nobody's asking for it, and letting it sit untouched for four months, wasn't building for users. It was building for comfort.

**The one question I can't fully answer**

Should I have written those 4,500 lines in the first place? Some of it, yes — I learned things building graph_memory.py that shaped how I understood LOOM. Some of it, probably not.

OpenTutor is open-source: https://github.com/zijinz456/opentutor
