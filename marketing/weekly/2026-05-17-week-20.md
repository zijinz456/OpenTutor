## Reddit Post — r/LocalLLaMA

**Title:** Built LLM-as-judge code grading for a local-first AI tutor — no execution, 132 lines, and one design decision I'm uncertain about

**Body:**

OpenTutor is an open-source learning platform that runs entirely on your own hardware — Ollama, LM Studio, or any OpenAI-compatible endpoint. I shipped a coding question type this week: the 8th question type in the quiz system, and the only one that can't be graded by a simple string comparison.

The approach: LLM-as-judge, no code execution. When a student submits code, the grader sends the question, the reference answer, and the student's code to whatever model you have configured. The system prompt tells the model to evaluate for functional equivalence, not surface matching: "Accept any correct implementation, not just the reference solution. Minor syntax variations (e.g. single vs double quotes, spacing) are fine. The logic must be functionally equivalent to the reference, not identical." The model returns `{"is_correct": true/false, "feedback": "brief reason"}`, feedback capped at 30 words. If the LLM call fails for any reason — connection error, timeout, parse failure — the grader falls back to `{"is_correct": false, "feedback": "Grading unavailable."}` and the submission still records.

The reason I chose no-execution: it keeps the attack surface at zero. Student code never runs server-side. There's no sandbox to escape, no resource exhaustion, no need to spin up isolated containers per submission. For a local tool running on one person's machine, this was an easy call.

The design decision I'm less confident about: without execution, the model might mark code correct that would fail on edge cases. A function that looks functionally equivalent might have an off-by-one error the LLM misses. For a low-stakes learning tool, I've decided the failure mode — a student gets positive reinforcement for slightly wrong code — is acceptable. I'd feel differently if this were a certification exam. But I'm genuinely uncertain whether that reasoning holds as more people use it.

The 132 lines break down as: the grader itself (63 lines), quiz parser registration (21 lines), annotation defaults for Bloom taxonomy tagging, submission router branching, and the frontend textarea component. The existing LOOM concept mastery updates, error classification, FSRS spaced repetition scheduling, and diagnostic pair auto-generation all work automatically — they key on `is_correct` regardless of how that was determined.

Tech stack: FastAPI, Next.js, SQLite, FSRS 4.5. Runs entirely locally.

Has anyone here used LLMs as judges for code correctness at any real scale? I want to know what failure modes you've actually observed — the confident wrong answer is what concerns me most.

## Reddit Post — r/Anki

**Title:** Added coding questions to a local-first tutor — LLM grades functional equivalence, no code runs

**Body:**

One gap I've always felt with Anki when studying programming: you can recognize the right answer in a multiple choice card, but that's very different from being able to write the code yourself. Recognition and recall are different skills. I've been building OpenTutor (open-source, runs on Ollama) and shipped a coding question type this week to try to close that gap.

How it works: when the system generates practice questions from your learning material, it can now produce coding questions alongside the usual multiple choice and true/false. The prompt for question generation — which runs against whatever LLM you have configured locally — is told to produce coding questions for programming and technical content: "write a function that does X", "complete this implementation", etc. When you submit your answer, the code goes to the LLM for grading. The comparison is semantic, not literal: the grader explicitly accepts any correct implementation, not just the reference solution. A list comprehension grades the same as a for loop if the outcome is equivalent.

The spaced repetition side (FSRS 4.5, same algorithm as newer Anki versions) works exactly as it does for other question types. A correctly-answered coding question gets its interval extended. A wrong answer resets the interval and, in the background, auto-generates a simpler diagnostic question to address the gap on the next session. That last part is something I've wanted in Anki for years — if I get a card wrong, give me a stepping-stone card, not just the same card again.

What's deliberately missing: no syntax highlighting, no language detection, no code execution. I wanted to ship the minimal version and see if anyone actually uses coding questions before building more around them.

The honest limitation: because nothing runs, the LLM might give positive feedback on code that fails at runtime. For a personal study tool, I think that tradeoff is acceptable. I'd reconsider for anything higher-stakes.

For programmers who use Anki — do you have any current strategy for testing code-writing rather than code-recognition? I'm curious what people do before something like this exists.

## IndieHackers Post

**Title:** Week 20: I let an LLM grade student code without running it. Here's the design tradeoff I'm not fully at peace with.

**Body:**

OpenTutor now supports coding questions. You write code in a textarea, hit submit, and the LLM you have configured locally decides if your answer is functionally correct. Nothing executes. No sandbox.

I've wanted this for a while. The quiz system has supported 7 question types since early on — multiple choice, true/false, short answer, fill in the blank, matching, select all that apply, free response. All of them either resolve to a string comparison or require a human judge. Coding was always the gap: you can't string-match code, and asking an LLM to grade every short answer response is expensive. But code is exactly the domain where LLM-as-judge makes sense — the model can reason about functional equivalence the way a human grader would.

The implementation took 132 lines across 5 files and one afternoon. The grader is 63 lines. The grading prompt tells the model to accept any correct implementation, not just the reference solution, and to cap its feedback at 30 words. If the LLM call fails — timeout, connection error, parse failure — the grader returns `is_correct: false` and "Grading unavailable." The submission still records. The student gets no credit for work they may have done correctly, but the system doesn't crash and the session continues. I think that's right for a best-effort feature.

The design I keep second-guessing: without executing the code, the LLM might mark something correct that would fail on edge cases. A function that looks right might have a subtle bug the model doesn't catch. The failure mode is a student getting positive reinforcement — and a spaced repetition interval bump — for code that doesn't fully work. That's a real problem. My current reasoning is: for a personal learning tool where the stakes are "did I understand this topic," that's acceptable. I'd feel very differently about this for a graded course or a certification system. But I don't have enough users yet to know whether my reasoning is right or just comfortable.

What I didn't ship this week alongside this: syntax highlighting (deliberate — I want to see adoption first), language detection, an inline display of the LLM's feedback reason. The feedback exists in the API response; I just haven't surfaced it properly in the UI yet. That's next if coding questions see any use.

The backend integration required almost no work because everything already keyed on `is_correct` — LOOM concept mastery, FSRS scheduling, error classification, prerequisite gap detection, diagnostic pair generation. The grader just substitutes for a string comparison. That felt like a good sign for the architecture.

OpenTutor is open-source and runs entirely locally on Ollama or any OpenAI-compatible API.

What's the risk threshold you use for "good enough" in a learning tool — when does "probably correct" become "close enough to ship"?
