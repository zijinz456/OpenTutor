## Reddit Post — r/LocalLLaMA

**Title:** I built a per-user cognitive load weight tuner for a local-first AI tutor — no ML, just a sliding window

**Body:**

I've been working on OpenTutor, an open-source AI learning platform that runs entirely on your own hardware (Ollama, LM Studio, or any OpenAI-compatible endpoint). One of its core features is a cognitive load detector that watches signals like error rate, message entropy, and session pace to decide when to intervene.

The problem I ran into: signal saturation. A student who spends 45 minutes drilling difficult flashcards will have an error_rate near 1.0 the entire time. The system kept treating that as "student is overloaded, intervene now" — but it's just what focused studying looks like for hard material. The signal was accurate but useless at those extremes.

**The fix I shipped this week:** a `cognitive_load_tuning.py` module, 136 lines. It maintains a per-user sliding window (size 20) of recent signal readings. If the mean for any signal exceeds 0.65 for long enough (minimum 5 readings), it multiplies that signal's weight by 0.90 — a small dampening step with a floor of 0.40 so the signal can never be zeroed out. When the mean falls back below threshold, weight recovers at 1.04× per reading to return toward 1.0.

The in-memory store is bounded to 200 users and evicts the least-active entry when full. I deliberately did not reach for a Bayesian update or any learned model. The comment in the code explains my reasoning: interpretability matters more than accuracy here. If the tuner is doing something weird to a user's score, I want to be able to explain it from first principles without running a model through its own internals.

The same commit also shipped two other things: a locked block overlay (blocks gated behind progression criteria now show a blurred overlay with an unlock hint instead of just not appearing) and a Tier-2 ambient agent action — the background monitor can now *propose* content generation (notes, quiz) but it creates a pending task the user must approve before anything runs. Dedup logic prevents stacking duplicate proposals.

Tech stack: FastAPI backend, Next.js frontend, SQLite, FSRS 4.5 for spaced repetition scheduling. Everything runs locally.

Happy to talk through the cognitive load architecture or the block unlock system in the comments. Is anyone else tackling signal saturation in adaptive systems? I'm curious whether per-user normalization is the right abstraction or whether I should be normalizing against a cohort.

## Reddit Post — r/Anki

**Title:** Built a tutor that stops crying wolf when you're just doing hard reviews — cognitive load auto-calibration

**Body:**

I use Anki heavily and one of my ongoing frustrations is that no AI tool I've tried actually knows the difference between "this student is struggling and needs help" and "this student is in flow on hard cards." They treat high error rate as always being a signal that something is wrong.

I've been building OpenTutor (open-source, local-first, runs on Ollama) and ran into exactly this problem. The system monitors cognitive load during study sessions by tracking error rate, message pacing, and a few other signals. But students who grind difficult material have persistently high error rates — that's just what the work looks like. Constantly flagging them for intervention breaks their flow.

This week I shipped a fix: a per-user sliding window that tracks each signal's recent mean. If error_rate stays above 0.65 for 20 consecutive readings, the tuner concludes this is probably the student's baseline, not a crisis, and gently reduces that signal's weight contribution. The weight recovers automatically when the signal returns to normal.

The practical effect: if you're deep in a hard Anki-style review session, the system stops treating your high error rate as an emergency and lets you keep going. It still monitors — the signal doesn't disappear — but it stops dominating the cognitive load score.

OpenTutor uses FSRS 4.5 for spaced repetition scheduling (same algorithm behind newer Anki versions). The learning workspace is block-based: flashcards, quiz, knowledge graph, and a few other blocks that unlock progressively as you study.

For anyone who's built adaptive learning tools — do you filter out sustained signals like this, or do you use a different approach to separate baseline from transient state? I went with the simplest thing that could work and I'm not sure it's the right call long-term.

## IndieHackers Post

**Title:** Week 16: My AI tutor was crying wolf and I fixed it with 136 lines of Python

**Body:**

There's a specific kind of bug that doesn't show up in tests: the system does exactly what you told it to do, and what you told it to do is subtly wrong.

OpenTutor monitors cognitive load during study sessions. It watches signals — error rate, session pace, message complexity — and uses them to decide whether to intervene, suggest a break, or back off. I built this over several months and was proud of it. Then I watched a real session.

A student spent 40 minutes grinding through hard physics flashcards. Error rate near 1.0 the entire time. The system kept queuing intervention suggestions every few minutes. The student ignored all of them and just kept studying. The system was technically correct — error rate was high — but it was missing the obvious: high error rate on genuinely hard material, sustained over time, is not an emergency. It's work.

The fix I shipped this week is called `cognitive_load_tuning.py`. It's 136 lines. It maintains a sliding window of 20 signal readings per user and applies a small dampening multiplier (0.90×, floored at 0.40) when any signal's mean stays elevated too long. When the signal drops back down, it recovers at 1.04× per reading. No machine learning. No model training. Just a bounded in-memory dict and some arithmetic.

I went back and forth on whether to reach for something more sophisticated — a Bayesian update, a small learned model per user. I decided against it. When the cognitive load system does something unexpected, I need to be able to explain it to a user in plain language. "Your error rate has been high for a while so I'm counting it less heavily" is something I can say. The output of a neural net is not.

The same week I also shipped a locked block overlay (blocks in the workspace now show a blurred overlay with an unlock hint when you haven't earned them yet — previously they just didn't appear, which was confusing) and a Tier-2 agent action for content proposals. The ambient background agent can now suggest generating notes or a quiz, but it creates a pending task you have to approve before anything runs. I added dedup so it doesn't stack the same proposal multiple times.

The honest challenge this week: none of this is validated yet. I don't have enough real users to know whether the tuning thresholds (0.65 mean, 0.90 decay) are the right numbers. They're defensible from first principles but they're guesses. The next thing I need to build is a way to observe the tuner's decisions in a dashboard so I can see if it's actually helping.

OpenTutor is open-source and runs entirely locally on Ollama or any OpenAI-compatible API. If you're building adaptive learning tools or have opinions on how to calibrate this kind of signal dampening, I'd genuinely like to hear them.

What's the hardest part of building something that's supposed to model a person's internal state?
