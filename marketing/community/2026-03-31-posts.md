(3) Three specialist agents (Tutor, Planner, Layout) routed by an intent orchestrator, rather than a single monolithic prompt. The codebase is ~370 Python files, 278 TypeScript files, 95 test files.

## b) r/LocalLLaMA

**Title:** `Built an adaptive study workspace that runs on llama3.2:3b — circuit breaker, multi-provider fallback, no cloud needed`

---

I've been building a local-first learning platform and wanted to share the LLM plumbing since it's a bit different from the usual "just call Ollama" approach.

**The setup:** Default is `LLM_PROVIDER=ollama` with `LLM_MODEL=llama3.2:3b`. That's it. No API key. The 3B model handles quiz generation, flashcard creation, and tutor chat well enough for most study material. For heavier reasoning (knowledge graph extraction, study plan generation) I use size-hint routing — agents can request a "large" or "fast" model variant and the router picks the right client:

```
LLM_MODEL=llama3.2:3b          # default
LLM_MODEL_LARGE=llama3.3:70b   # for complex reasoning
LLM_MODEL_FAST=llama3.2:1b     # for low-latency interactions
```

**The circuit breaker:** The router (`apps/api/services/llm/router.py`) wraps every provider in a `CircuitBreakerMixin`. After 3 consecutive failures, the circuit opens and the provider is bypassed for 120 seconds. Progressive cooldowns are [5, 10, 20, 60] seconds before that. If the primary goes down, it automatically tries the next healthy provider in the fallback chain. Background health probes run every 30 seconds to update the cache.

**Supported providers out of the box:** Ollama, vLLM, LM Studio, TextGen WebUI, OpenAI, Anthropic, DeepSeek, Gemini, Groq, OpenRouter, LiteLLM, or any OpenAI-compatible endpoint. Switch by changing two env vars.

**RAM usage with llama3.2:3b (Q4_K_M):** ~2.4 GB. The app stack itself (FastAPI + Next.js + SQLite) is another ~300 MB. Runs fine on 8 GB machines.

**What doesn't work well at 3B:** multi-hop knowledge graph extraction and long study plan generation. That's why the agent system explicitly routes those to the "large" variant if configured. Without it, it falls back gracefully with a simpler output.

Repo: https://github.com/zijinz456/OpenTutor — 3 commands to run locally.

## c) r/Anki

**Title:** `I implemented FSRS 5/6 from scratch for my local AI tutor — here's what's different from what Anki ships`

---

I've been running Anki since 2018 and switched to FSRS as soon as it shipped. When I built my own local study tool, I wanted to implement the algorithm myself to understand it properly. Here's what I learned.

**SM-2 (what Anki used until recently):**
- Fixed ease factor, adjusted by ±0.1 per rating
- No model of forgetting curve shape — just a multiplier on the previous interval
- Ease hell: one bad run of reviews can tank your ease factor permanently

**FSRS-4.5 (what Anki ships now):**
- 17 parameters, optimized per-user
- Models stability (days until 90% recall) separately from difficulty
- Retrievability formula: `R(t) = (1 + t/S)^(-1)` — actual forgetting curve

**FSRS 5/6 (what I implemented):**
- 21 parameters (added same-day review handling and trainable decay)
- Retrievability: `R(t) = (1 + factor·t/S)^(-decay)` where decay is a trainable parameter (defaults to 1.0 for backward compatibility). When `decay = 1.0` it's identical to FSRS-4.5.
- Same-day review formula: `S' = S * exp(w17 * (G - 3 + w18)) * S^(-w19)` — prevents stability inflation when you review a card multiple times in one session
- Difficulty initialization: `D0 = w4 - exp(w5 * (G - 1)) + 1` (exponential, not linear like 4.5)

**What else I added on top:**
- A `forgetting_cost` metric that estimates how many cards you're expected to lose right now, not just counting overdue cards. Review urgency goes: none → low → normal → high → critical based on expected loss, not card count.
- Integration with a knowledge graph (LOOM) for prerequisite-aware review ordering — so if you're weak on "osmosis," it surfaces "membrane permeability" first.

The full FSRS implementation is 304 lines in `apps/api/services/spaced_repetition/fsrs.py`. The claimed accuracy improvement over SM-2 is 30%+ (from the open-spaced-repetition benchmarks).

For anyone curious, the repo is https://github.com/zijinz456/OpenTutor — it runs entirely locally with Ollama, no Anki replacement intended, just a different workflow for course material.

## e) Dev.to / Hashnode — Technical Article

**Title:** `How I built a multi-provider LLM router with circuit breakers for my AI tutor`

---

*Tags: python, llm, ai, opensource, architecture*

---

When I started building OpenTutor — a local-first adaptive learning platform — I assumed the LLM layer would be the boring part. Pick a provider, call the API, done. Six months later, the LLM router is one of the more interesting pieces of the system: 399 lines of Python handling provider registration, health monitoring, circuit breaking, size-hint routing, and automatic fallback. Here's how it's structured and why.

### The problem with naive provider switching

The first version was simple: read `LLM_PROVIDER` from env, instantiate one client, use it everywhere. This broke in three ways:

1. **Ollama timeouts during model loading** — the first request after a cold start could take 30+ seconds, and a timeout would mark the request as failed with no retry logic.
2. **API key misconfiguration** — a user would set `LLM_PROVIDER=openai` with a wrong key, every request would 401, and the app would keep trying instead of surfacing the error.
3. **Agent size preferences** — the Tutor agent for flashcard chat wants a fast small model; the Planner agent generating a 4-week study schedule wants a larger one. A single client couldn't express this.

### The circuit breaker

Each provider client mixes in `CircuitBreakerMixin` from `apps/api/services/llm/circuit_breaker.py`:

```python
# Progressive cooldown steps (borrowed from openakita)
COOLDOWN_STEPS = [5, 10, 20, 60]  # seconds

# Circuit breaker thresholds
CIRCUIT_OPEN_THRESHOLD = 3   # failures before opening circuit
CIRCUIT_RESET_TIMEOUT = 120  # seconds before trying again

class CircuitBreakerMixin:
    def mark_unhealthy(self, error: str):
        """Progressive cooldown + circuit breaker."""
        self._healthy = False
        self._consecutive_failures += 1

        if self._consecutive_failures >= CIRCUIT_OPEN_THRESHOLD:
            self._circuit_open = True
            self._circuit_open_time = time.time()
            logger.error(
                f"Circuit OPEN for {self.provider_name} after "
                f"{self._consecutive_failures} failures: {error}"
            )
            return

        idx = min(self._consecutive_failures - 1, len(COOLDOWN_STEPS) - 1)
        cooldown = COOLDOWN_STEPS[idx]
        self._cooldown_until = time.time() + cooldown
```

First failure → 5s cooldown. Second → 10s. Third → circuit opens, provider bypassed for 120s, then auto-reset. The `is_healthy` property handles the auto-reset check on every read.

### The provider registry

Providers register into a `ProviderRegistry` at startup. The registry is built declaratively from a table of provider definitions:

```python
_OPENAI_COMPAT_PROVIDERS = [
    ("openai",     "openai_api_key",     None,                                                         "gpt-4o-mini",             ("gpt-", "o1-", "o3-", "o4-")),
    ("deepseek",   "deepseek_api_key",   "https://api.deepseek.com/v1",                                "deepseek-chat",           ("deepseek-",)),
    ("gemini",     "gemini_api_key",     "https://generativelanguage.googleapis.com/v1beta/openai/",   "gemini-2.0-flash",        ("gemini-",)),
    ("groq",       "groq_api_key",       "https://api.groq.com/openai/v1",                             "llama-3.3-70b-versatile", ("llama-", "mixtral-", "gemma-")),
    # ... OpenRouter, etc.
]
```

All OpenAI-compatible providers share the same `OpenAIClient` class — just different `base_url` and `api_key`. Adding a new provider is one line in the table. Ollama, vLLM, LM Studio, and TextGen WebUI all use this same path.

The `.get()` method walks a fallback chain: requested provider → model variant → primary → next healthy provider. If everything is down, it raises `RuntimeError` with a clear message instead of silently returning garbage.

### Size-hint routing for agents

The three specialist agents (Tutor, Planner, Layout) have different model preferences. Rather than hardcoding model names in each agent, they request a hint:

```python
# Agent requests a fast model for low-latency chat
client = get_llm_client("fast")

# Agent requests a large model for complex reasoning
client = get_llm_client("large")
```

The registry resolves hints to concrete clients registered at startup:

```python
for attr, hint in [("llm_model_large", "large"), ("llm_model_small", "small"),
                   ("llm_model_fast", "fast"), ("llm_model_standard", "standard"),
                   ("llm_model_frontier", "frontier")]:
    variant_model = getattr(settings, attr, None)
    if variant_model and variant_model != model:
        _register_variant(registry, provider, variant_model, hint)
```

If `LLM_MODEL_LARGE` isn't set, `get_llm_client("large")` falls through to the primary. No config required for single-model setups.

### Background health monitoring

A background task probes every registered provider every 30 seconds and caches latency + health status:

```python
async def _probe_one(self, name: str, client: LLMClient) -> dict:
    start = time.time()
    try:
        healthy = await client.ping()
        elapsed = (time.time() - start) * 1000
        return {
            "healthy": healthy,
            "status": "ok" if healthy else "unhealthy",
            "latency_ms": round(elapsed, 1),
            "checked_at": time.time(),
        }
    except Exception as e:
        ...
```

This powers a `/api/llm/health` endpoint that the frontend uses to show provider status without adding latency to actual requests.

### What I'd do differently

The registry is initialized lazily as a global singleton with a threading lock (`_registry_lock`). It works, but it means the first request pays the initialization cost. A better approach would be to initialize at app startup in the FastAPI lifespan handler and inject via dependency. That's on the roadmap.

The full implementation is in [`apps/api/services/llm/`](https://github.com/zijinz456/OpenTutor/tree/main/apps/api/services/llm) — router.py (399 lines), circuit_breaker.py (83 lines), and provider clients in `providers/`. The project is MIT licensed and open to contributions.

If you're building anything with local LLMs and want a pattern that doesn't melt when Ollama is loading a model, this approach has held up well through 6 months of daily use.
