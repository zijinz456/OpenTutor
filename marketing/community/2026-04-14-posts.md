## b) r/LocalLLaMA

**Title:** `LLM-graded coding questions in my local study app — what actually works at 3B vs 7B vs 70B`

---

I just shipped an 8th question type to my open-source learning platform: coding questions graded by the local LLM instead of an execution sandbox. No Docker-in-Docker, no isolated runtime — the model reads the submitted code, compares it to the expected behavior, and returns a pass/fail with feedback. Here's what I've learned about model capability at different sizes:

**llama3.2:3b (Q4_K_M, ~2.4 GB):**
Good at: detecting obviously wrong code, basic Python/JS syntax errors, simple off-by-one identification. Struggles with: anything requiring multi-step reasoning about algorithm correctness, edge case analysis, time complexity evaluation. For "write a function that reverses a string" — reliable. For "implement a BFS with cycle detection" — hit or miss.

**llama3.2:7b / mistral-7b (Q4_K_M, ~4.5 GB):**
Noticeably better at partial credit reasoning and explaining *why* code is wrong, not just that it is. Works well for undergraduate-level CS quiz content.

**llama3.3:70b (Q4_K_M, ~40 GB):**
This is where LLM grading becomes genuinely trustworthy for coding. Grade agreement with human rubrics is high enough that I'm comfortable using it for exam prep material.

The router handles this with size hints — the coding grader explicitly requests the "large" model variant:

```
LLM_MODEL=llama3.2:3b          # default (fast questions)
LLM_MODEL_LARGE=llama3.3:70b   # coding grader, study plan generation
```

If `LLM_MODEL_LARGE` isn't set, it falls back to primary. No config required to get started — you just get less reliable coding grades at 3B.

The fallback also matters here: if Ollama is mid-load (loading the 70B weights), the circuit breaker's 5s→10s→20s progressive cooldown keeps the grader from hammering a busy server. After 3 failures, it opens for 120s and tries the next healthy provider.

The app itself (FastAPI + Next.js + SQLite) runs at ~300 MB RAM. Total with llama3.2:3b: ~2.7 GB. Tight but works on an 8 GB machine.

Repo: https://github.com/zijinz456/OpenTutor — 3 commands to run.

## c) r/Anki

**Title:** `I added a difficulty selector on top of FSRS — uses Vygotsky's ZPD to pick question hardness based on mastery`

---

I've been running FSRS in my own study tool for a while and recently shipped something that Anki doesn't do: the quiz difficulty adapts based on your BKT mastery score for each concept, not just the FSRS interval.

Here's the problem FSRS alone doesn't solve: it tells you *when* to review something, but not *how hard* the question should be. If you've mastered a concept at 80%, you don't need recall questions anymore — you need edge cases and traps. If you're at 35%, you need fundamentals before anything else.

I implemented a 3-layer difficulty system based on Vygotsky's ZPD:

```
P(L) < 0.40  → Layer 1: recall / fundamental (70% L1, 30% L2, 0% L3)
0.40–0.70    → Layer 2: application / transfer (20% L1, 50% L2, 30% L3)
P(L) ≥ 0.70  → Layer 3: traps / edge cases (10% L1, 30% L2, 60% L3)
```

There are also three gap-type overrides that bypass the mastery score entirely:
- `fundamental_gap`: forces Layer 1 regardless of P(L) — you know the procedure but missed a prerequisite
- `transfer_gap`: caps at Layer 2 — you've memorized it but can't apply it
- `trap_vulnerability`: pushes to Layer 3 — you understand it deeply but keep falling for a specific error pattern

BKT tracks these at the concept level. FSRS handles the *when*, BKT handles the *what difficulty*. They're orthogonal and both run simultaneously per flashcard.

There's also a `relearning` state from FSRS that overrides everything and forces Layer 1 — if a card has lapsed back to relearning, you don't need trap questions, you need to re-establish the foundation first.

The implementation is 166 lines in `apps/api/services/learning_science/difficulty_selector.py`. The BKT model is in `knowledge_tracer.py` (same directory). The combined FSRS+BKT scheduler lives in `apps/api/services/spaced_repetition/`.

For anyone curious: https://github.com/zijinz456/OpenTutor — local-first, runs with Ollama, not trying to replace Anki, just a different workflow for course material with an AI tutor attached.

Above 70% means the system is throwing edge cases and traps at you.

It also added a "summary block" — a daily digest that shows what you reviewed this session, what you're predicted to forget in the next 48 hours (using FSRS retention forecasting), and what the AI tutor thinks your biggest gap is right now. Takes about 10 seconds to read but saves 20 minutes of confused "what should I study tonight" paralysis.

The forgetting forecast is the part I didn't expect to use but now check every morning. It shows estimated retention for each topic based on how long ago you last reviewed it. Topics dropping below 70% predicted retention get flagged. It's not perfect — local LLMs are not GPT-4 — but the scheduler (FSRS algorithm, same one Anki switched to) is solid and the predictions have been accurate enough to trust.

If you're doing boards prep or finals and you don't have a clear visibility into your mastery by topic — not just "I did 50 flashcards" but "my mastery score for this specific concept is 62% and trending up" — it's worth trying. ## e) Dev.to / Hashnode — Technical Article

**Title:** `How I built a 12-signal cognitive load detector that adapts my AI tutor in real time`

---

*Tags: python, ai, machinelearning, education, opensource*

---

The hardest part of building an AI tutor isn't generating good explanations. It's knowing when to *stop* generating complex explanations and switch to something simpler. A student who's been studying for 90 minutes, just got three questions wrong in a row, and is sending two-word messages ("I don't understand") is in a different cognitive state than a student who just started and is writing detailed questions. Treating them the same way is one of the most common failure modes in AI tutoring systems.

For [OpenTutor](https://github.com/zijinz456/OpenTutor), I built a 12-signal behavioral cognitive load detector that runs on every chat message and injects adaptation guidance into the tutor's system prompt. Here's how it works.

### What cognitive load detection actually means

I'm not doing anything exotic. There's no EEG or eye tracking. I'm computing a weighted sum of behavioral proxies — things you can observe from the interaction log — that correlate with high cognitive load. The output is a score from 0.0 to 1.0 and a level (low/medium/high) that the tutor agent uses to decide how to respond.

The theoretical basis is a 2025 Nature paper: ["Cognitive Load Theory meets Deep Knowledge Tracing"](https://www.nature.com/articles/s41598-025-10497-x) — it validated several of the behavioral signals I'm using in a classroom setting.

### The 12 signals

Each signal is normalized to [0.0, 1.0] and multiplied by a configurable weight. Here's the full breakdown from `apps/api/services/cognitive_load.py` (488 lines):

```python
# Signal 1: Text-based fatigue (pre-computed keyword analysis)
signals["fatigue"] = fatigue_score
load += fatigue_score * settings.cognitive_load_weight_fatigue

# Signal 2: Session length — cognitive performance drops after ~45 min
session_fatigue = min(session_messages / settings.cognitive_load_session_messages_norm, 1.0)
signals["session_length"] = session_fatigue
load += session_fatigue * settings.cognitive_load_weight_session_length

# Signal 5: Help-seeking language — pre-compiled regex patterns
_HELP_PATTERNS = [
    re.compile(p) for p in [
        r"\bhelp\b", r"\bhint\b", r"\bexplain\b", r"\bdon'?t understand\b",
        r"\bconfused\b", r"\bstuck\b", r"\bhow do i\b",
    ]
]
help_signal = 1.0 if any(p.search(_msg_lower) for p in _HELP_PATTERNS) else 0.0

# Signal 7: Answer hesitation time (median of last 10 quiz responses)
if len(times) >= 3:
    median_ms = statistics.median(times)
    timing_signal = min(max(
        (median_ms - settings.cognitive_load_hesitation_min_ms)
        / settings.cognitive_load_hesitation_range_ms, 0.0), 1.0)
    signals["answer_hesitation"] = timing_signal
```

The remaining signals are: unmastered error count, message brevity (short messages = possible frustration), quiz accuracy vs. target, NLP-based affect analysis (frustration + confusion scored by LLM), per-user relative baseline calibration, wrong-answer streak, inter-message pause duration (30s–300s range), and repeated errors on the same problem.

Signal 8 — NLP affect — is the expensive one. It calls the LLM to score frustration and confusion in the student's message. To avoid adding 500ms to every response, I only run it every 3 messages and reuse the cached result in between:

```python
# LLM affect call is expensive (~500ms). Only call every 3 messages;
# reuse the last result for intermediate messages.
_should_run_nlp = (session_messages <= 1) or (session_messages % 3 == 0)
if _should_run_nlp and user_message:
    affect = await analyze_student_affect(user_message)
    async with _last_affect_lock:
        _last_affect[str(user_id)] = affect
else:
    async with _last_affect_lock:
        affect = _last_affect.get(str(user_id), {})
```

Signal 9 — relative baseline calibration — deserves its own explanation. Raw signal values aren't comparable across students. A student who always writes short messages isn't more cognitively loaded than one who writes long ones — they just communicate differently. The baseline tracker builds a per-user rolling average of message length and help-seeking frequency, then expresses each new message as a deviation from their own baseline. A message is only flagged as a brevity signal if it's unusually short *for that person*.

### What the tutor does with it

The output of `compute_cognitive_load()` is a `guidance` string injected into the tutor's system prompt. For high load with 3+ consecutive high-load messages, it looks like this:

```
## Cognitive Load: HIGH (score: 0.74)

The student is showing signs of cognitive overload:
- active help-seeking language detected
- message brevity below personal baseline (deviation: -0.42)
- answer hesitation above median (median: 8,200ms)
- 3+ consecutive high-load messages

Adapt your response:
• Use shorter sentences and simpler vocabulary
• Break explanations into numbered steps
• Offer one concept at a time
• Consider asking the student what specifically is confusing (Socratic)
• Avoid introducing new concepts until current confusion resolves
```

The tutor doesn't see the raw numbers — just the adaptive instruction. This keeps the system prompt readable and lets me change the detection logic without rewriting prompts.

### The weight auto-tuner

The trickiest part is that default signal weights don't work equally well for every student. I recently shipped a Phase 1 auto-tuner: a sliding window that tracks whether high-load predictions from the last session correlated with the student accepting an intervention (simplification, hint, break suggestion) or ignoring it. Signals that predicted accepted interventions get a small multiplier boost; signals that predicted ignored interventions decay slightly. The multipliers are bounded at [0.5, 2.0] to prevent runaway calibration.

```python
# ── Phase 1 weight auto-tuning: update sliding window ──
try:
    from services.cognitive_load_tuning import update_and_get_multipliers
    weight_multipliers = update_and_get_multipliers(str(user_id), signals)
except Exception:
    weight_multipliers = {}
```

It's still early-stage — the data collection window needs a few weeks of real usage before the multipliers stabilize — but the pattern is in place.

### What I'd do differently

The biggest limitation is that all 12 signals are *indirect*. I'm inferring cognitive load from behavior rather than measuring it. Some students are naturally terse. Some have high frustration tolerance. The baseline calibration helps but doesn't fully solve this. If I were starting over, I'd instrument more timing data earlier — specifically, time-to-first-keypress after a question is displayed, which is one of the better proxies for cognitive load in the literature and requires zero NLP.

The full implementation is in [`apps/api/services/cognitive_load.py`](https://github.com/zijinz456/OpenTutor/blob/main/apps/api/services/cognitive_load.py) — 488 lines, MIT licensed. The project also includes BKT-based mastery tracking, FSRS 4.5 spaced repetition, a 3-agent system (Tutor, Planner, Layout), and runs fully locally with Ollama.

If you're building anything adaptive with local LLMs and have questions about the signal design, happy to discuss in the comments.
