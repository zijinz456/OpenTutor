## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – study blocks unlock progressively based on behavior, not time`

---

Most educational tools dump every feature on you at launch. OpenTutor has 12 learning blocks — notes, quiz, flashcards, knowledge graph, study plan, LECTOR review, wrong-answer tracker, cognitive forecast, and more. A new user sees five of them. The rest unlock as you use the product. I want to explain the reasoning and the implementation because the tradeoff is non-obvious.

The unlock logic lives in `apps/web/src/lib/block-system/feature-unlock.ts` (74 lines). Five blocks are always available: notes, quiz, flashcards, chapter list, and the basic review queue. Beyond that: the knowledge graph unlocks after 3+ source documents are uploaded — it needs enough content to build a meaningful concept graph. The wrong-answers block unlocks after your first incorrect answer — there's nothing useful to show before that. The AI forecast block unlocks after 50+ practice attempts — below that, the underlying FSRS model doesn't have enough signal to predict forgetting curves meaningfully. The agent insight block (which surfaces behavioral patterns across sessions) unlocks after 3 sessions — a single session produces noise, not signal.

```typescript
case "knowledge_graph":
  return ctx.sourceDocCount >= 3
    ? { unlocked: true }
    : { unlocked: false, reason: "Unlocks after uploading 3+ source documents",
        unlockHint: `${ctx.sourceDocCount}/3 documents` };

case "forecast":
  return ctx.practiceAttempts >= 50
    ? { unlocked: true }
    : { unlocked: false, reason: "Unlocks after 50+ practice attempts",
        unlockHint: `${ctx.practiceAttempts}/50 attempts` };
```

Unlock state is tracked in localStorage per course — no server round-trips. The conditions are evaluated client-side on every render. The tradeoff we made: features that sound impressive but require context to be useful (knowledge graph, cognitive load forecast) are hidden until that context exists. Showing "knowledge graph" on day one when you have one PDF and zero practice attempts would be misleading — the graph is nearly empty, the forecast is meaningless, and you'd dismiss both as useless before they've had a chance to work.

The counterargument we considered: users should know what they're getting. Our answer is the unlock hint shown in the UI — a progress indicator that explains exactly what triggers the unlock. You always know what's coming. You just don't get it until the precondition is real.

Repo is 357 Python + 269 TypeScript files, MIT licensed. Runs locally on Ollama with `docker compose up -d --build`.

GitHub: https://github.com/zijinz456/OpenTutor

## b) r/LocalLLaMA

**Title:** `I built a 3-tier model router for a study app — how we avoid burning the GPU on greetings`

---

OpenTutor is an open-source AI learning platform I'm building. It routes requests to different local model sizes based on a complexity score — no LLM call to decide, just a scoring function that runs in under 1ms. Here's the design.

**The problem**

Running everything through your largest local model is wasteful. A student typing "ok" or "what's the next topic?" doesn't need a 13B parameter model. A student asking "prove why this sorting algorithm is O(n log n) and derive the recurrence" genuinely does. On a single machine with Ollama, routing poorly means either burning GPU cycles on simple requests or using a model too small for complex ones.

**Three tiers**

`apps/api/services/llm/complexity.py` (196 lines) scores each request and routes to one of three model tiers:

- **fast**: Simple queries, greetings, preference changes (`score < 150`)
- **standard**: Teaching, exercises, review — most education tasks (`150 ≤ score < 500`)
- **frontier**: Multi-step reasoning, code grading, complex planning (`score ≥ 500`)

**The scoring function**

Five inputs, no LLM involved:

```python
def score_complexity(
    message: str,
    intent: str = "general",   # classified upstream: "learn", "quiz", "plan", etc.
    scene: str = "study_session",
    history_length: int = 0,
    has_rag_context: bool = False,
) -> int:
    score = 0
    score += _score_message_length(message)       # 0-150 based on char count
    score += _score_intent(intent)                 # intent "plan" → 250, "learn" → 150
    score += _score_scene(scene)                   # "exam_prep" → +80
    score += _score_conversation_depth(history_length)  # deeper threads → higher score
    score += _score_complexity_markers(message)    # regex: "prove", "derive", "step-by-step"
    if has_rag_context:
        score += 50                                # synthesis bonus
    return score
```

Complexity markers include CJK patterns too (证明/推导 → +50, 分析/解析 → +30), since we have Chinese-language users. The intent classifier runs first and is itself a lightweight call — "general" and "preference" intents never reach frontier tier regardless of score.

**Agent minimum tiers**

Some agents always need a minimum model regardless of message complexity:

```python
AGENT_MIN_TIERS = {
    "preference": ModelTier.FAST,
    "teaching":   ModelTier.STANDARD,
    "planning":   ModelTier.FRONTIER,   # multi-week curriculum generation
    "code_execution": ModelTier.FRONTIER,  # LLM-graded coding questions
}
```

`resolve_tier()` returns `max(scored_tier, agent_min_tier)` — the agent floor wins if score alone would route it lower.

**In practice with Ollama**

You can wire three variants — say llama3.2:1b for fast, llama3.2:3b for standard, and a 7B+ for frontier — using `register_variant()` in the provider registry. The score-to-tier routing happens before the Ollama request is made. Cold start on llama3.2:3b is ~8s, ~0.8s/response warm. Fast requests go to 1b and come back in ~0.3s warm. Frontier stays in queue until the bigger model is loaded.

MIT licensed, runs with `docker compose up -d --build`.

## c) r/Anki

**Title:** `I upgraded from FSRS-4.5 to FSRS-5/6 — here's what actually changed in the math`

---

I've been implementing spaced repetition from scratch for an open-source study platform (OpenTutor), and recently upgraded the scheduler from FSRS-4.5 to FSRS-5/6. I wanted to share what concretely changed because I've seen a lot of posts that say "5 is more accurate" without explaining why.

**Parameter count: 17 → 21**

FSRS-4.5 has 17 parameters (w0–w16). FSRS-5/6 adds four more (w17–w20). Three of the new params (w17, w18, w19) handle same-day reviews. The fourth (w20) is the forgetting curve decay, which was hardcoded at 1.0 in 4.5 and is now trainable.

**Forgetting curve: fixed → trainable**

FSRS-4.5's retrievability formula is essentially:
`R(t) = (1 + t/9S)^(-1)` — a fixed power-law decay.

FSRS-5/6 makes the exponent trainable:
`R(t) = (1 + factor·t/S)^(-decay)` where `decay = w[20]`

When `w[20] = 1.0` (the default), you get the FSRS-4.5 formula exactly. The benefit of making it trainable: the optimizer can fit your actual forgetting curve to real review data rather than assuming a universal exponent.

**Same-day reviews: stability inflation fixed**

In FSRS-4.5, doing 5 reviews of the same card in one day would inflate stability significantly — each review applied the standard stability formula. FSRS-5/6 adds a separate formula for intra-day reviews:

`S' = S · exp(w17 · (G - 3 + w18)) · S^(-w19)`

When `w17 = w18 = w19 = 0` (defaults I ship), the formula degenerates to `S' = S` — no change, but also no inflation. Non-zero params let it model same-day learning curves specifically.

**Difficulty initialization: linear → exponential**

FSRS-4.5: `D0 = w4 - (G - 3) · w5` — linear in rating.
FSRS-5/6: `D0 = w4 - exp(w5 · (G - 1)) + 1` — exponential.

The difference is that the exponential version makes it much harder to get to difficulty 1 (easiest) by rating "Easy" on first review, which matches observed behavior better — new cards that are trivially easy on first exposure still tend to have some difficulty over time.

**LECTOR: knowledge-graph-aware scheduling on top**

One thing I added that Anki doesn't do: the review queue isn't just FSRS-ordered. It's influenced by the knowledge graph. If a prerequisite concept is weak (`mastery < 0.4`), the dependent concept gets boosted in priority — you review the prereq first. If two concepts are flagged as "confused_with" edges in the graph, they get reviewed in "contrast" mode together. This is from the LECTOR paper (arxiv:2508.03275).

Standard FSRS is card-level. LECTOR operates at concept-level and uses FSRS retrievability scores as one factor in a multi-signal priority ranking.

Happy to share the implementation if anyone is curious — the scheduler is `apps/api/services/spaced_repetition/fsrs.py` (304 lines) and LECTOR is `apps/api/services/lector.py` (344 lines). MIT licensed.

GitHub: https://github.com/zijinz456/OpenTutor

## d) r/GetStudying / r/medicalschool

**Title:** `The study mistake I kept making: jumping into new material before checking I'd actually retained the prerequisites`

---

I want to share something that shifted how I study, because I don't see it discussed as concretely as it deserves.

When I was preparing for exams I kept doing the same thing: I'd finish Chapter 4, feel like I understood it, and move straight to Chapter 5. Two weeks later, Chapter 5 concepts weren't clicking — not because Chapter 5 was hard, but because I'd started to forget Chapter 4, which Chapter 5 was built on. I was building on sand.

The mistake is obvious in hindsight. Prerequisite knowledge decays just like everything else. But we tend to treat understanding as binary (got it / didn't get it) rather than probabilistic. You understood Chapter 4 *at the time*. That doesn't mean you still understand it now.

**What I changed**

Before starting a new topic, I do a 5-minute "prerequisite check" instead of diving in. I write down 3 specific things Chapter N+1 depends on from Chapter N, and I quiz myself on them — not re-reading, actually trying to recall. If I miss even one, I spend 10 minutes reviewing before moving forward.

It feels slow. It actually saves time. Hitting a wall 4 chapters later and having to re-trace where the gap is costs way more than 10 minutes upfront.

**Tracking wrong answers by concept, not by question**

The other thing that helped: I stopped tracking "which questions I got wrong" and started tracking "which concepts I'm uncertain about." A question can be wrong for four different reasons — maybe I got the formula right but the underlying assumption wrong, or I misread the question. What matters is the concept that broke down.

Going back through wrong answers with the question "what was the underlying concept I didn't actually understand?" is a different and more useful exercise than just re-doing the same questions.

**When you're genuinely stuck**

If I'm stuck on a concept and reviewing it isn't helping, the most useful question I've found is: *"What would I need to know first to understand this?"* Usually the answer surfaces a gap one or two steps back. The concept I'm stuck on isn't the real problem — the missing prerequisite is.

I've been using an open-source study tool I'm building (OpenTutor) that does this automatically — it tracks prerequisite relationships between concepts and promotes the weakest ones before moving forward. But the principle itself doesn't require any tooling. A pencil and a list works just as well.

What study patterns have you found for catching prerequisite gaps early?

## e) Dev.to / Hashnode — Technical Article

**Title:** How I built a 12-signal behavioral sensor to detect when students are struggling

---

The standard way to adapt AI tutoring difficulty is to watch quiz performance. High accuracy → increase difficulty. Low accuracy → decrease it. This works, but it's late: by the time quiz performance drops, the student has already failed several questions and may be frustrated.

I wanted earlier signals — behavioral indicators that a student is overloaded *before* they've gotten multiple things wrong. This led to a 12-signal cognitive load detector in OpenTutor, a local-first AI learning platform I'm building. Here's how it works.

### Why behavioral signals instead of just performance metrics

The core insight from the Cognitive Load + DKT paper (Nature Scientific Reports, 2025) is that cognitive overload shows up in behavior before it shows up in wrong answers. Short messages, long pauses, repeated help requests, and decreasing answer latency are all detectable in real time. Quiz performance is the outcome; these signals are the leading indicators.

The system is in `apps/api/services/cognitive_load.py` (488 lines). Every time a student sends a message, `compute_cognitive_load()` runs and returns a score from 0.0 to 1.0 that gets injected into the tutor's system prompt to adjust explanation depth, vocabulary, and question difficulty.

### The 12 signals

**1. Text-based fatigue** — pre-computed by an upstream module that classifies message tone. Passed in as `fatigue_score`.

**2. Session length fatigue** — cognitive performance degrades after ~45 minutes. This signal increases linearly as the session gets longer:

```python
session_fatigue = min(session_messages / settings.cognitive_load_session_messages_norm, 1.0)
```

The normalization constant is configurable. Default behavior: a 45-message session contributes full session fatigue.

**3. Unmastered error count** — wrong answers that haven't been corrected yet. Five or more unmastered errors drives this signal to maximum:

```python
error_signal = min(recent_errors / settings.cognitive_load_error_count_norm, 1.0)
```

**4. Message brevity** — short messages correlate with frustration. A one-word message after several paragraphs of explanation suggests something broke down:

```python
brevity_signal = max(0.0, 1.0 - (msg_len / settings.cognitive_load_brevity_length_norm))
```

Only active after the first message — a one-word greeting shouldn't trigger cognitive overload detection.

**5. Help-seeking keywords** — explicit requests signal the student has already tried and failed. Seven patterns compiled at startup:

```python
_HELP_PATTERNS = [
    re.compile(p) for p in [
        r"\bhelp\b", r"\bhint\b", r"\bexplain\b", r"\bdon'?t understand\b",
        r"\bconfused\b", r"\bstuck\b", r"\bhow do i\b",
    ]
]
```

**6. Recent quiz performance** — accuracy below the target threshold contributes proportionally:

```python
perf_signal = max(0.0, 1.0 - (accuracy / settings.cognitive_load_quiz_accuracy_target))
```

**7. Answer hesitation** — quiz response timing. Median of the last 10 quiz answers. A median above the hesitation threshold contributes to load:

```python
timing_signal = min(max(
    (median_ms - settings.cognitive_load_hesitation_min_ms)
    / settings.cognitive_load_hesitation_range_ms, 0.0), 1.0)
```

This captures slow answers without being thrown off by outliers (median, not mean).

**8. NLP affect analysis** — a lightweight LLM call that classifies frustration and confusion from the message text. This is the most expensive signal (~500ms), so it only runs every 3 messages; the cached result is reused for the intermediate 2. The frustration and confusion scores are weighted and added to the total.

**9. Relative baseline calibration** — each student has different natural message lengths and help-seeking rates. Signal 4 and 5 would be misleading if one student always writes one-word messages and another always asks for hints. A per-user baseline tracked in SQLite normalizes signals relative to the student's own behavior, not a global average.

**10. Wrong answer streak** — 3 or more consecutive wrong answers is a strong overload signal independent of the running error count:

```python
wrong_streak_signal = min(streak / 5.0, 1.0)   # 5+ consecutive = max
```

**11. Message gap / pause hesitation** — long pauses between messages (30–300 seconds) indicate the student is stuck. Pauses under 30s are normal thinking time; over 300s likely indicates a break. The meaningful signal is in between:

```python
if 30 < gap_seconds < 300:
    gap_signal = min((gap_seconds - 30) / 270.0, 1.0)
```

**12. Repeated errors on the same problem** — getting the same question wrong twice without mastering it is a qualitatively different signal than getting different questions wrong. This detects deep confusion vs. scattered gaps:

```python
# Count problems with 2+ wrong attempts still unmastered
repeated_signal = min(repeated_count / 3.0, 1.0)   # 3+ repeated problems = max
```

### Combining signals into a score

Each signal contributes a weighted fraction to the final score. Weights are configurable in settings — the default balance prioritizes NLP affect and error streak most heavily, since those are the most reliably predictive of genuine overload rather than normal study friction. The total is clamped to `[0.0, 1.0]`.

```python
score = max(0.0, min(load, 1.0))
```

The score maps to three levels — `"low"`, `"medium"`, `"high"` — with a corresponding guidance fragment injected into the tutor's system prompt. High load: simplify explanations, reduce vocabulary complexity, skip extension questions. Low load: increase depth, introduce edge cases, ask synthesis questions.

### What this actually changes about the tutor

The score injection is a single paragraph prepended to the system prompt:

- **Low**: "The student appears ready for challenging material. Introduce nuance and edge cases."
- **Medium**: "The student may be approaching capacity. Use clear, concrete explanations. Avoid multiple concepts at once."
- **High**: "The student is showing overload signals. Simplify. Use short sentences. Ask only one question at a time. Avoid new vocabulary."

The workspace also reacts: the block decision engine's `rule_cognitive_overload` fires when `score >= 0.7` and removes non-essential blocks (knowledge graph, forecast, agent insight) to reduce visual complexity. The inverse rule restores them when `score < 0.4` for two consecutive turns.

### Performance

The NLP affect call (Signal 8) is the bottleneck at ~500ms every 3 messages. The other 11 signals are database queries — mostly indexed lookups — and take under 20ms combined. The full function runs async, so it doesn't block the tutor's response generation. The result is cached in-memory per user and written to a `cognitive_load_score` column in the chat session record for analytics.

**Stack:** FastAPI, Python 3.11+, SQLAlchemy 2 async, SQLite, Next.js 16. Runs locally with Ollama (llama3.2:3b default). 372 backend test functions, MIT licensed.

Full repo: https://github.com/zijinz456/OpenTutor — the cognitive load module is `apps/api/services/cognitive_load.py`. The block decision engine that consumes the score is `apps/api/services/block_decision/engine.py`. Feedback welcome.
