## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – FSRS spaced repetition that decays across a knowledge graph`

---

Most flashcard tools treat cards in isolation. You review "mitochondria" without the system knowing whether you've also forgotten "ATP synthesis" — the upstream concept that makes mitochondria meaningful. We shipped something that changes this, and I want to describe the technical decision because it's not obvious.

The system now runs two algorithms in tandem. FSRS (upgraded to 5/6 with 21 parameters — up from 4.5's 17) schedules cards by retrievability. A separate module called LECTOR (based on arxiv:2508.03275) reads the LOOM knowledge graph (arxiv:2511.21037) and reorders the review queue based on three additional signals: prerequisite decay propagation (if concept A depends on concept B and B's stability falls below threshold, A gets a review boost), confusion pair clustering (concepts that the student historically confuses get reviewed as contrast pairs), and prerequisite priority (a weak prerequisite surfaces before its dependents, regardless of FSRS score). The ordering logic is in `apps/api/services/lector.py` (344 lines) and graph traversal in `apps/api/services/loom_graph.py` (428 lines). The FSRS scheduler itself is `apps/api/services/spaced_repetition/fsrs.py` (304 lines, pure Python, no dependencies on the graph layer).

The other thing I want to flag: the cognitive load detector now has a sliding-window auto-tuner. There are 12 behavioral signals (session duration, error rate, message brevity, help-seeking language, quiz accuracy, response timing, NLP affect, baseline calibration, wrong-answer streak, inter-message pause, repeated errors, text fatigue). Each signal has a configurable weight. The tuner maintains a 20-sample window per user and dampens weights when a signal is persistently elevated — the assumption being that a chronic high-error-rate is baseline noise, not acute overload. The decay factor is 0.90 per tuning step, floored at 0.40. Code: `apps/api/services/cognitive_load_tuning.py` (136 lines). Repo is 357 Python + 269 TypeScript/TSX files, MIT, runs with `docker compose up -d --build` against Ollama — no API key required.

Genuine question: has anyone compared LECTOR-style prerequisite-aware scheduling against vanilla FSRS on a controlled study cohort? The arxiv paper has simulation results but no RCT. Would be interested in collaborators who want to run something.

GitHub: https://github.com/zijinz456/OpenTutor

## b) r/LocalLLaMA

**Title:** `I built a study app that runs entirely on llama3.2:3b — here's what breaks and what doesn't`

---

I've been running OpenTutor (open-source, self-hosted learning platform) almost exclusively on `llama3.2:3b` via Ollama for the past few months and wanted to share an honest breakdown of what works at 3B and where you genuinely need a larger model.

**What works fine at 3B:**
- Structured note generation from uploaded PDFs/PPTX — consistent JSON output with prompt engineering
- Flashcard generation — short Q/A pairs, handles well
- MCQ and true/false quiz generation — works reliably
- Grounded chat Q&A (RAG) on factual material — good enough for most study content
- Study plan suggestions and deadline reasoning

**What degrades at 3B:**
- Coding question grading — pass/fail judgment on nontrivial algorithms is unreliable; 7B+ is noticeably better
- Short-answer evaluation — the rubric-following gets loose; tends to over-credit partial answers
- Knowledge graph extraction (LOOM) — the concept-relationship extraction from dense academic text produces more noise edges; works better at 13B+
- Socratic questioning chains — the model loses the thread after 3-4 exchanges

**The infrastructure side:** The LLM router (`apps/api/services/llm/router.py`, 399 lines) has a circuit breaker with progressive cooldowns: 5s → 10s → 20s → 60s, opening the circuit after 3 consecutive failures with a 120-second reset timeout. This matters for local Ollama because the model can take 30-60s to load on first call — the circuit breaker distinguishes "still loading" from "actually dead" by probing with a background health monitor every 30 seconds. You can register multiple providers and it automatically falls back down the chain.

**RAM usage in practice (M2 MacBook, 16GB):**
- llama3.2:3b Q4: ~2.5GB, ~1.2s/response for short prompts
- llama3.1:8b Q4: ~5.5GB, ~2.8s/response
- The app itself (FastAPI + Next.js + SQLite): ~400MB

**Setup is 3 commands:**
```bash
git clone https://github.com/zijinz456/OpenTutor.git && cd OpenTutor
cp .env.example .env  # set LLM_PROVIDER=ollama, LLM_MODEL=llama3.2:3b
```

## c) r/Anki

**Title:** `I implemented FSRS from scratch (upgraded to 5/6) in my study app — here's what's different from Anki's version`

---

I've spent the last few months implementing and iterating on FSRS for an open-source study platform I'm building, and I wanted to share some notes for people who care about the algorithm details — particularly what's different between FSRS 4.5, FSRS 5/6, and what Anki ships.

**FSRS 4.5 vs FSRS 5/6 — the parameter difference that matters most**

Anki's FSRS implementation (as of the current stable release) uses FSRS 4.5 with 17 parameters. FSRS 5/6 adds 4 more (21 total). The additions I care about most:

1. **Trainable forgetting curve decay (w20):** FSRS 4.5 uses a fixed power-law forgetting curve. FSRS 5/6 makes the decay exponent a parameter: `R(t) = (1 + factor·t/S)^(-decay)` where `decay = w[20]`. When `decay = 1.0`, it's backward-compatible with 4.5. The upside: the curve can fit individual memory profiles better. The downside: you need enough review history to train it meaningfully.

2. **Same-day review stability formula (w17-w19):** FSRS 4.5 has a known issue — reviewing the same card multiple times in one day inflates stability. FSRS 5/6 adds a dedicated formula: `S' = S * exp(w17 * (G - 3 + w18)) * S^(-w19)` to dampen intra-day inflation. This matters a lot if you do multiple review passes per day.

3. **Exponential difficulty initialization:** FSRS 4.5 uses linear init (`D0 = w4 - (G-3)*w5`). FSRS 5/6 uses exponential (`D0 = w4 - exp(w5*(G-1)) + 1`), which distributes initial difficulty more naturally across the rating scale.

**What I added that Anki doesn't have: LECTOR semantic prioritization**

The standard FSRS queue is purely time-based. I added a layer on top (based on [arxiv:2508.03275](https://arxiv.org/abs/2508.03275)) that reorders the queue using a knowledge graph:

- If concept B is a prerequisite for concept A, and B's retrievability drops below 0.7, it gets promoted ahead of A — even if A is "due" first by FSRS timing
- Concepts you historically confuse get grouped for contrast review
- When you forget one concept, related concepts in the graph get a proactive review boost before they technically "expire"

The queue ordering code is about 344 lines. The forgetting cost estimator (decides how urgent a session is) is modeled after Orbit's session batching algorithm: it sums `(1 - R)` across all overdue cards and triggers "critical" urgency when the expected forgotten count >= 5.

**Comparison summary:**

| Feature | Anki FSRS | OpenTutor |
|---|---|---|
| Algorithm version | FSRS 4.5 (17 params) | FSRS 5/6 (21 params) |
| Same-day dampening | No | Yes |
| Decay trainability | Fixed | Trainable |
| Knowledge-graph ordering | No | Yes (LECTOR) |
| Forgetting cost estimator | No | Yes |
| Platform | Cross-platform | Self-hosted, local-first |

Repo: https://github.com/zijinz456/OpenTutor — the FSRS implementation is `apps/api/services/spaced_repetition/fsrs.py` if you want to read the actual math.

## d) r/GetStudying / r/medicalschool

**Title:** `The thing that actually helped me retain material: making my review sessions fight each other`

---

I want to share a study technique I've been using that's been more effective than anything else I've tried for dense content like physiology and pharmacology.

The problem I kept running into: I'd review a flashcard on "loop diuretics mechanism" and get it right, feel good, move on. Then a week later I'd blank on *why* furosemide causes hypokalemia — not because I forgot the card, but because I'd been reviewing the card in isolation from the concept it depended on (Na-K-ATPase, aldosterone response).

The fix that worked for me: I started doing contrast reviews. I'd deliberately pull up two cards that I knew I confused — loop diuretics and thiazides, for example — and review them back-to-back specifically to force the comparison. Wrong answer on one? Review it right next to the confusable one, not in isolation.

Manually, this is annoying to set up. I ended up building it into an open-source tool I've been working on (OpenTutor, self-hosted, runs locally) — the review queue now automatically groups concepts you historically confuse and surfaces them as contrast pairs. It also does prerequisite ordering: if your understanding of aldosterone is weak, it bumps that card before anything that depends on it, rather than waiting for the FSRS algorithm to schedule it.

For practical study tips that don't require any tooling:

1. **After any wrong answer, immediately find the related concept you might have confused it with and review both.** Don't just re-review the one you got wrong.
2. **Build a "confusion map" — a simple list of pairs you mix up.** Review from this list separately from your normal rotation.
3. **Before reviewing a topic, ask yourself: what does this concept depend on?** Review prerequisites first, even if they're not "due."
4. **Track stability, not just correctness.** A card you got right three times in a row is different from one you got right once after a month gap.

The research behind this is the LECTOR framework (arxiv paper from 2025) on semantic spaced repetition — it formalizes what good tutors intuitively do: they don't just test you on what's due, they sequence reviews based on conceptual dependencies.

Good luck with your studies. Feel free to ask if you want more on the contrast-pair technique.

## e) Dev.to / Hashnode

**Title:** How I built a 12-signal cognitive load detector that adapts my AI tutor's teaching style in real time

---

When I started building OpenTutor — a local-first AI learning platform — I made a mistake that most AI tutoring tools make: I treated all learner messages the same. Same explanation depth, same pacing, same level of Socratic questioning, regardless of whether the student was fresh and focused or 90 minutes into a session and clearly struggling.

The fix was a real-time cognitive load detector. Here's exactly how it works.

### The Problem with a Single-Mode Tutor

The tutor agent in OpenTutor generates responses by injecting a system prompt with behavioral guidance. Initially that prompt was static. The result: when students were fatigued, the tutor kept pushing the same depth of explanation. When they were breezing through easy content, it stayed in "simplified" mode. Both are wrong.

I needed the tutor to *read* the student and adjust. The challenge: we have no physiological sensors. We only have behavioral traces — message text, timing, quiz performance, session length.

### The 12 Signals

The detector (`apps/api/services/cognitive_load.py`, 488 lines) computes a score from 0.0 to 1.0 using 12 behavioral signals, each with a configurable weight:

```python
async def compute_cognitive_load(
    db: AsyncSession,
    user_id: uuid.UUID,
    course_id: uuid.UUID,
    fatigue_score: float = 0.0,
    session_messages: int = 0,
    user_message: str = "",
) -> dict:
    signals: dict[str, float] = {}
    load = 0.0

    # Signal 1: Text-based fatigue (pre-computed from message NLP)
    signals["fatigue"] = fatigue_score
    load += fatigue_score * settings.cognitive_load_weight_fatigue

    # Signal 2: Session length — cognitive perf degrades after ~45 min
    session_fatigue = min(session_messages / settings.cognitive_load_session_messages_norm, 1.0)
    signals["session_length"] = session_fatigue
    load += session_fatigue * settings.cognitive_load_weight_session_length
```

The full list of signals:

| # | Signal | What it measures |
|---|---|---|
| 1 | Text fatigue | NLP pre-analysis of fatigue cues in message text |
| 2 | Session length | Normalized count of messages (proxy for time-on-task) |
| 3 | Unmastered error count | DB query: unmastered wrong answers for this course |
| 4 | Message brevity | Message length — terse messages often indicate frustration |
| 5 | Help-seeking language | Pattern match: "help", "confused", "don't understand", etc. |
| 6 | Quiz accuracy | Recent correct/attempts ratio from learning progress table |
| 7 | Answer hesitation | Time between question display and answer submission |
| 8 | NLP affect | LLM-based analysis of emotional tone (frustration/confusion) |
| 9 | Relative baseline | Per-user calibration against their own historical baseline |
| 10 | Wrong-answer streak | Consecutive incorrect answers in current session |
| 11 | Message gap | Inter-message pause time — long pauses suggest difficulty |
| 12 | Repeated errors | Same question failed multiple times |

### The Circuit Breaker for Inference Calls

Signal 8 (NLP affect) requires an LLM call. This creates a problem: if your local Ollama is under load, that signal can block or error. The LLM router has a circuit breaker that handles this:

```python
# apps/api/services/llm/circuit_breaker.py
COOLDOWN_STEPS = [5, 10, 20, 60]   # progressive backoff in seconds
CIRCUIT_OPEN_THRESHOLD = 3          # failures before opening circuit
CIRCUIT_RESET_TIMEOUT = 120         # seconds before auto-reset

def mark_unhealthy(self, error: str):
    self._consecutive_failures += 1
    if self._consecutive_failures >= CIRCUIT_OPEN_THRESHOLD:
        self._circuit_open = True
        self._circuit_open_time = time.time()
        return  # Circuit is open — skip future calls until reset

    idx = min(self._consecutive_failures - 1, len(COOLDOWN_STEPS) - 1)
    cooldown = COOLDOWN_STEPS[idx]
    self._cooldown_until = time.time() + cooldown
```

When the circuit opens, the affect signal defaults to 0.0, and the remaining 11 signals carry the load score. This degrades gracefully — you lose one signal, not the whole system.

### The Auto-Tuner: Preventing Signal Saturation

The hardest problem I ran into: a student who consistently has low quiz accuracy (because the material is genuinely hard) would have `unmastered_errors` near 1.0 for every session. This saturates that signal and prevents the detector from responding to *changes* in cognitive load.

The fix: a sliding-window weight tuner (`apps/api/services/cognitive_load_tuning.py`, 136 lines):

```python
WINDOW_SIZE = 20
HIGH_SIGNAL_THRESHOLD = 0.65
WEIGHT_DECAY_FACTOR = 0.90    # reduce weight when signal is chronically high
MIN_WEIGHT_RATIO = 0.40       # floor — don't suppress entirely
RECOVERY_FACTOR = 1.04        # nudge back up when signal drops

def update_and_get_multipliers(user_id: str, signals: dict[str, float]) -> dict[str, float]:
    state = _get_or_create(user_id)
    multipliers = {}
    for signal_name, value in signals.items():
        dq = state.windows.setdefault(signal_name, collections.deque(maxlen=WINDOW_SIZE))
        dq.append(value)
        mean = sum(dq) / len(dq)
        current = state.multipliers.get(signal_name, 1.0)
        if mean >= HIGH_SIGNAL_THRESHOLD:
            new_mult = max(current * WEIGHT_DECAY_FACTOR, MIN_WEIGHT_RATIO)
        else:
            new_mult = min(current * RECOVERY_FACTOR, MAX_WEIGHT_RATIO)
        state.multipliers[signal_name] = new_mult
        multipliers[signal_name] = new_mult
    return multipliers
```

The intuition: if `unmastered_errors` is high for 20 messages straight, it's probably baseline for this student, not an acute signal. Dampen it. When it drops below threshold, gradually restore weight. No ML — just a sliding window mean with bounded multipliers.

### Feeding the Score into the Tutor Prompt

The output is a guidance fragment injected into the tutor's system prompt:

```python
# level: "low" | "medium" | "high"
if level == "high":
    guidance = (
        "The student appears cognitively overloaded. "
        "Use shorter sentences, simpler vocabulary, and one concept at a time. "
        "Offer hints rather than full explanations. Consider Socratic prompting."
    )
elif level == "medium":
    guidance = (
        "The student may be experiencing some difficulty. "
        "Check comprehension before advancing. Keep explanations concrete."
    )
else:
    guidance = "The student appears engaged and ready. Normal explanation depth."
```

This is blunt but effective. The tutor doesn't know *why* the load is high — it just adapts depth. More nuanced interventions (switching to Socratic mode, suggesting a break, adjusting quiz difficulty) come from a separate ambient monitor agent.

### What I'd Do Differently

The biggest open problem: we're computing cognitive load per-message, but the weights are tuned offline. A proper implementation would run an A/B experiment: hold weights fixed for a control group, apply auto-tuning for a treatment group, and measure outcomes (quiz accuracy improvement, session completion rate). We haven't done this yet because we're still in single-user local-first beta.

If you're building something similar, the key insight is: **behavioral signals are noisy individually but surprisingly useful in ensemble**. Even simple signals like message brevity and inter-message pause carry real information about student state. The trick is preventing any single signal from dominating.
