## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – adaptive workspace driven by a deterministic rule engine, not an LLM`

---

Most "AI-adaptive" tools mean: call the LLM, ask it what to do next, inject the answer somewhere. We do this for teaching, but not for workspace layout. Every time the workspace reshapes itself in OpenTutor — adding a review block because forgetting risk is rising, hiding the knowledge graph when cognitive load spikes, restoring it when load recovers — that's a 257-line deterministic rule engine with zero LLM calls. I want to explain why we made that choice.

The system is `apps/api/services/block_decision/engine.py`. Thirteen rules evaluate independently on each turn: `rule_cognitive_overload`, `rule_cognitive_recovery`, `rule_forgetting_risk`, `rule_frustration`, `rule_deadline_approaching`, `rule_lector_review`, `rule_mastery_gate`, `rule_confusion_pairs`, `rule_prerequisite_gap`, `rule_weak_areas`, `rule_inactivity`, `rule_mastery_complete`, `rule_cognitive_adapt`. Each returns a list of `BlockOperation` dataclasses with `(action, block_type, reason, urgency, config)`. The engine collects all operations, sorts by urgency descending, enforces `MAX_OPS_PER_TURN = 2`, and filters against a dismiss history. That's the entire decision loop — no LLM, no neural net, no fuzzy logic. The rule for cognitive overload, for example, fires when `score >= 0.7` and `consecutive_high >= 2`, then removes up to 3 non-protected blocks in priority order: `["agent_insight", "forecast", "knowledge_graph", "progress", "plan", "wrong_answers", "flashcards", "review"]`. The inverse rule (`rule_cognitive_recovery`) restores removed blocks when `score < 0.4` and `consecutive < 2`. Notes, quiz, and chapter list are in a `PROTECTED_BLOCKS` frozenset and can never be removed.

The reason to avoid LLM here is boring and correct: latency, reliability, and testability. An LLM-based workspace controller adds 1–5 seconds to every message turn, fails silently when the local model is loading, and produces outputs you can't snapshot-test. The rule engine is synchronous, ms-latency, and every rule has a discrete test case. The tradeoff is expressiveness — the rule engine can't reason about novel situations it wasn't written for. We accept that. The LLM handles teaching; the rule engine handles the workspace. Clean separation. Repo is 357 Python + 269 TypeScript files, 996 test functions, MIT licensed. Runs locally with `docker compose up -d --build` and Ollama — no API key.

GitHub: https://github.com/zijinz456/OpenTutor

## b) r/LocalLLaMA

**Title:** `I built a 10-provider LLM router for a local study app — here's how the fallback chain actually works`

---

OpenTutor is an open-source AI learning platform I've been building. It supports 10+ LLM providers and routes between them automatically when one fails. I want to share how the provider system is wired because it's the part that makes running on local hardware actually reliable.

**The ProviderRegistry**

The core is a `ProviderRegistry` class in `apps/api/services/llm/router.py` (399 lines total). You register providers with a name and a priority flag:

```python
registry.register("ollama_primary", ollama_client, primary=True)
registry.register("ollama_fallback", ollama_fallback_client)
registry.register("openai", openai_client)
```

When `registry.get()` is called, it walks the `_fallback_order` list and returns the first healthy provider. "Healthy" is determined by the circuit breaker mixin — each provider tracks its own state independently.

**The circuit breaker**

`apps/api/services/llm/circuit_breaker.py` (83 lines). Progressive cooldown + hard circuit open:

```python
COOLDOWN_STEPS = [5, 10, 20, 60]   # seconds per consecutive failure
CIRCUIT_OPEN_THRESHOLD = 3          # failures before hard-open
CIRCUIT_RESET_TIMEOUT = 120         # seconds before auto-reset
```

Failure 1 → 5s cooldown. Failure 2 → 10s. Failure 3 → circuit opens, all requests for this provider skip for 120 seconds, then auto-probe. The reason for the open circuit vs. continued cooldowns: when Ollama is loading a 7B model, it can take 30–60s. A cooldown-based system keeps retrying during that window and hammering the process. Circuit open + auto-reset is cleaner — you wait, then do one probe, then re-add to fallback rotation.

**Model size routing**

You can register model "variants" for size hints:

```python
registry.register_variant("small", llama3b_client)
registry.register_variant("large", llama70b_client)
registry.register_variant("fast", llama3b_client)
```

Internal code that needs a faster/cheaper call (like the cognitive load affect signal, which fires every 3 messages) requests `"small"`. Heavy tasks like knowledge graph extraction request `"large"`. This lets you wire one machine with two local models and route between them by task weight without any manual configuration per call site.

**What this costs in practice**

The router + circuit breaker adds maybe 0.5ms of overhead per call. The background health probe runs on a 30-second interval and writes results into `_probe_cache` in-memory — no DB. Cold start latency on Ollama is the actual bottleneck: llama3.2:3b takes ~8s to load on a cold container, ~0.8s/response once warm. The fallback chain to a cloud provider (e.g., DeepSeek at ~0.3s) means you're never stuck waiting for a local model cold start if you don't want to be.

Config is all in `.env`:

```bash
LLM_PROVIDER=ollama
LLM_MODEL=llama3.2:3b
# Optional fallback:
LLM_FALLBACK_PROVIDER=deepseek
LLM_FALLBACK_MODEL=deepseek-chat
DEEPSEEK_API_KEY=sk-...
```

MIT licensed. `docker compose up -d --build` and it's running. Happy to answer questions about the Ollama integration or the circuit breaker design.

## c) r/Anki

**Title:** `I built an AI study app that exports directly to Anki .apkg — here's why I kept both`

---

I've been building OpenTutor, an open-source local-first study platform, and one of the first features users asked for was Anki export. The reasoning was usually: "I already have Anki decks, I just want the AI content generation and then I'll review in Anki."

So I built it: the agent can export flashcards to `.apkg` format on request. You just ask the tutor "export my flashcards to Anki" and it runs the export tool and returns a download link. The export handles batch IDs (so you can export a specific card set from one upload) or dumps the whole course.

But here's what I've actually found from using both: they serve different learning phases well, and the answer isn't to pick one.

**Where OpenTutor is better**

- Content generation: upload a PDF, get 40 cards in 30 seconds, already organized by topic. Anki's card creation is manually intensive.
- Active learning during ingestion: AI tutor grounded in the source material, Socratic questioning, 7 question types including coding questions graded by the LLM.
- Prerequisite-aware review (LECTOR): cards get promoted/demoted in the queue based on whether their upstream concepts are stable. Standard Anki FSRS doesn't do this.
- Cognitive load adaptation: under high load (detected from behavioral signals), easier cards surface first. Anki doesn't adapt in session.

**Where Anki is better**

- Cross-platform mobile review. OpenTutor's mobile layout is still a known gap.
- AnkiDroid sync, AnkiWeb backup, the mature ecosystem.
- Custom card templates, add-ons, the community.
- If you have existing decks you've maintained for years, you're not rebuilding them.

**The workflow I ended up with**

Use OpenTutor to generate and initially learn material — the active AI tutoring during the learning phase is genuinely useful. Once a card is stable enough that you just need periodic review, export it to Anki and let AnkiDroid handle the phone reviews. The FSRS stability scores don't transfer (the formats don't align cleanly), but the card content does — you're essentially graduating cards from active learning to passive maintenance.

OpenTutor runs locally on Ollama (llama3.2:3b works fine for card generation), no API key required, MIT licensed.

Has anyone else found a good Anki + local AI workflow? I'm curious how people are using them together.

## d) r/GetStudying / r/medicalschool

**Title:** `The thing that actually helped my exam prep: an AI that tells me when to *stop*, not just what to study`

---

I want to share something that shifted how I think about study sessions, because I don't see it talked about much.

Most study advice is about what to study, how to study, which techniques to use. Almost none of it is about when to stop — or more specifically, when to switch from active learning to something lower-intensity, and when to just close the laptop.

I've been using an open-source study tool I've been building (OpenTutor, runs locally) that has what I call an ambient monitor — a background agent that watches your session and occasionally surfaces suggestions. Not pop-ups, not notifications every few minutes. It has a rule baked in: "prefer silence over spam. If the student is on track, do nothing. Never send more than 1 notification per 6 hours per course."

What it actually flags:

**"You've been in this session for a while and haven't generated practice questions yet"** — useful when I get lost reading notes without ever testing myself.

**"You have 5 concepts at high forgetting risk from last week's session"** — prompts a quick review before I go deeper into new material.

**"You seem to be struggling right now, and your deadline is 3 days away — consider switching to a lower-pressure mode"** — this one caught me off guard the first time it fired. It had noticed I'd been getting quiz questions wrong repeatedly and my messages were getting shorter (a frustration signal), and it suggested switching to "self-paced" review instead of "exam prep" intensity.

The last one specifically helped me. I was cramming pharmacology the week before a shelf exam and getting increasingly frustrated — the app noticed before I did that I needed to back off. Taking 20 minutes to do lower-stakes flashcard review instead of trying to hammer high-difficulty questions actually helped me consolidate better.

**Practical tips that don't require any tooling:**

1. Set a timer for 45 minutes. At the end, ask yourself: "am I understanding more, or am I just going through motions?" If it's the second, stop.
2. Before starting a session, write down what you're worried about forgetting, not what you want to learn. Review those things first.
3. Track your wrong answers separately from your review pile. Wrong answers need a different intervention than "just review it again."
4. Notice when your messages or notes are getting shorter/less specific. That's usually a signal you're running low, not a sign you've mastered the material.

The ambient monitoring is just a formalization of what a good study partner would do: watch for when you're grinding unproductively and say something.

## e) Dev.to / Hashnode — Technical Article

**Title:** How I replaced "use Socratic method" in a prompt with an actual finite-state machine

---

When I started building the tutor agent in OpenTutor — a local-first AI learning platform — I made the mistake that most AI tutoring systems make. I wrote a system prompt that said something like: "Use the Socratic method. Don't give away answers. Ask guiding questions."

It worked sometimes. More often, the LLM would ask one Socratic question and then, if the student gave even a vague answer, immediately provide the full explanation anyway. The instruction was aspirational, not structural. There was no mechanism to maintain Socratic mode across turns, adapt based on what kind of confusion the student showed, or eventually break out of it when a student was genuinely stuck.

The fix was to stop treating "use Socratic method" as a prompt instruction and start treating it as a program. Here's how I built a 5-state FSM that drives the Socratic teaching strategy in `apps/api/services/agent/socratic_engine.py` (287 lines).

### The five states

```python
class SocraticState(str, Enum):
    PROBE    = "probe"     # Ask open-ended question — student reasons aloud
    CLARIFY  = "clarify"   # Narrow the question — student was vague
    CONFRONT = "confront"  # Present counterexample — student has misconception
    SCAFFOLD = "scaffold"  # Give one hint — student is stuck
    CONFIRM  = "confirm"   # Transfer question — student seems to understand
```

Each state maps to a specific directive injected into the tutor's system prompt for that turn. Not a full prompt — just a fragment that overwrites the teaching strategy section:

```python
_DIRECTIVES: dict[SocraticState, str] = {
    SocraticState.PROBE: (
        "Ask the student an open-ended question about this concept. "
        "Do NOT give the answer. Let them reason. "
        "Example: 'What do you think happens when...?'"
    ),
    SocraticState.CONFRONT: (
        "The student has a misconception. Present a counterexample "
        "that challenges their current understanding. "
        "Do NOT say 'you're wrong'. Let the counterexample create cognitive conflict. "
        "Example: 'If that were true, what would happen in this case...?'"
    ),
    SocraticState.SCAFFOLD: (
        "The student needs help. Give ONE small hint that moves them forward "
        "without giving the full answer. Break the problem into a smaller step. "
        "Example: 'Let's start with just the first part. What is X?'"
    ),
    # ...
}
```

The LLM never decides which mode to be in — it's told which mode to be in. The FSM makes the decision.

### The transition table

Every turn, after the student responds, the engine classifies the response quality as one of: `"correct"`, `"partial"`, `"wrong"`, `"confused"`, `"no_response"`. Then a static transition table maps `(current_state, response_quality) → next_state`:

```python
_TRANSITIONS: dict[tuple[SocraticState, str], SocraticState] = {
    (SocraticState.PROBE, "correct"):     SocraticState.CONFIRM,
    (SocraticState.PROBE, "partial"):     SocraticState.CLARIFY,
    (SocraticState.PROBE, "wrong"):       SocraticState.CONFRONT,
    (SocraticState.PROBE, "confused"):    SocraticState.SCAFFOLD,
    (SocraticState.PROBE, "no_response"): SocraticState.SCAFFOLD,

    (SocraticState.CONFRONT, "correct"): SocraticState.CONFIRM,
    (SocraticState.CONFRONT, "partial"): SocraticState.CLARIFY,
    (SocraticState.CONFRONT, "wrong"):   SocraticState.SCAFFOLD,
    # ... 20 total transitions
}
```

No LLM decides the transition. No fuzzy logic. If the student gives a partial answer to a clarifying question, they go to SCAFFOLD. Always. That determinism is the point — the Socratic strategy is now testable in isolation from the LLM.

### Initial state from student context

The FSM doesn't always start at PROBE. The initial state is computed from three inputs available at the start of a conversation:

```python
def _initial_state(self) -> SocraticState:
    if self.cognitive_load > 0.7:
        return SocraticState.SCAFFOLD   # Overloaded student → start with hints
    if self.mastery < 0.3:
        return SocraticState.SCAFFOLD   # Low mastery → scaffold first
    if self.mastery > 0.7:
        return SocraticState.PROBE      # High mastery → challenge them
    if self.error_type == "conceptual":
        return SocraticState.CONFRONT   # Known misconception → counterexample
    if self.error_type in ("procedural", "computational"):
        return SocraticState.SCAFFOLD   # Procedural confusion → step-by-step
    return SocraticState.CLARIFY        # Default: narrow the question
```

This means the Socratic engine is context-aware from the first message. A student who just got a conceptual question wrong starts in CONFRONT mode. A cognitively overloaded student (detected by the 12-signal cognitive load system) skips PROBE entirely and starts with scaffolding. Neither of these require an LLM decision — the context comes from the database and the cognitive load score.

### The scaffold ceiling

There's one guard I added that the pure FSM doesn't handle: a maximum number of turns in SCAFFOLD before giving up and just explaining directly:

```python
_MAX_SCAFFOLD_TURNS = 4

def get_prompt_directive(self) -> str:
    if self.state == SocraticState.SCAFFOLD and self.turns_in_state >= _MAX_SCAFFOLD_TURNS:
        return (
            "The student has been stuck for several turns. "
            "Stop hinting. Provide a clear, direct explanation now."
        )
    return _DIRECTIVES[self.state]
```

This was the most important addition. Pure Socratic method can be cruel — if a student genuinely doesn't have the background knowledge, making them guess indefinitely is worse than explaining. Four scaffold turns without progress and the engine breaks out of the pattern and tells the LLM to just teach.

### State persistence

Socratic state is persisted per `(user_id, course_id)` in a lightweight key-value store (backed by a `agent_kv` table in SQLite). The engine serializes to/from a plain dict:

```python
def to_dict(self) -> dict:
    return {
        "state": self.state.value,
        "turns_in_state": self.turns_in_state,
        "mastery": self.mastery,
        "cognitive_load": self.cognitive_load,
        "error_type": self.error_type,
    }
```

This means Socratic mode survives session restarts. If you left a session mid-SCAFFOLD, you come back mid-SCAFFOLD. The engine doesn't restart at PROBE when you return.

### Why FSM over prompt?

Prompt-based strategies are stateless. Each LLM call is independent — the model has no memory of which Socratic state it was in two turns ago unless you re-inject that into context. By tracking state externally and injecting a single directive per turn, you get:

- **Deterministic, testable transitions**: you can unit-test every `(state, response_quality) → state` pair without an LLM.
- **Auditability**: you can log which state fired on which turn and trace exactly why the tutor asked what it asked.
- **Composition with other signals**: cognitive load and mastery influence initial state and scaffold ceiling. The FSM is a clean place to combine multiple signals without the LLM having to reason about all of them simultaneously.
- **Controllable complexity**: the LLM handles *content* (what to say in CONFRONT mode, given this specific misconception). The FSM handles *strategy* (whether to be in CONFRONT mode at all). Separation of concerns.

The system isn't perfect — response quality classification is still done by the LLM (a lightweight call asking "was this response correct, partial, wrong, confused, or no response"), and that's the one step where the FSM depends on model reliability. But the strategy itself is code, not vibes.
