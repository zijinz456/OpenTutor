## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – feature blocks that unlock based on learner behavior`

---

Most adaptive learning tools front-load every feature — the student opens the app and gets 12 panels, 4 sidebars, and an onboarding wizard. The usual alternative is subscription gating, which doesn't apply to open-source. I tried a third option: behavioral unlocks tied to whether a feature has enough data to be meaningful.

OpenTutor is a local-first AI learning platform (FastAPI + Next.js 16, runs on Ollama). The workspace is 12 composable blocks — notes, quiz, flashcards, knowledge graph, review queue, forecast, weak spots, daily digest, study plan, and more. Most of them start locked. Unlock conditions are derived from what the student actually does, not how long they've been a user:

- **Knowledge Graph**: 3+ source documents (a graph from one PDF is trivially small)
- **Forecast**: 50+ practice attempts (fewer makes trajectory prediction noise, not signal)
- **Weak Spots**: first wrong answer (empty before then — nothing to surface)
- **Agent Insight**: 3+ sessions (the model needs behavioral history to generate useful workspace suggestions)

The implementation is a pure `isBlockUnlocked(type, context)` function (~75 lines, `apps/web/src/lib/block-system/feature-unlock.ts`). The context object — `{sourceDocCount, practiceAttempts, hasWrongAnswer, sessionCount, ...}` — is derived from localStorage, updated during normal app usage. No backend round-trip for unlock state.

The locked overlay renders the component behind frosted glass — not hidden. Students see a blurred preview of what they're unlocking. The overlay shows a progress counter: "2/3 documents" or "47/50 attempts". This communicates value before access and gives a concrete next action. The rest of the stack: FSRS 4.5 spaced repetition, LOOM knowledge graph, LECTOR semantic review prioritization, 3-agent system (Tutor, Planner, Layout). 996 test functions, MIT, `docker compose up -d --build`.

GitHub: https://github.com/zijinz456/OpenTutor

## b) r/LocalLLaMA

**Title:** `I built a two-tier LLM gate for local content generation — the 3B model decides, the bigger model acts`

---

OpenTutor is an open-source AI learning platform (Ollama-first, llama3.2:3b by default). I want to share a routing pattern we just shipped for a problem that's specific to local LLMs: **content generation latency**.

The app has an ambient study monitor — an LLM-driven background agent that watches your learning state and decides whether to intervene. One of its available actions is `propose_content_generation`: the monitor detects that you've studied for a while but have no AI-generated notes or quiz questions yet, and proposes creating them.

This is a **Tier-2 action** — it creates a pending task in the workspace that requires explicit user approval. Nothing generates until you approve. The reason:

1. Content generation from a 10-page PDF takes 15–35s on a 3B model depending on hardware. Firing it silently creates a jarring UI stall when results arrive mid-session.
2. The monitor's inference about *what* you need (notes vs. quiz) can be wrong. The approval card lets you redirect before committing.
3. Local-first apps need an explicit boundary between "model observes" and "model acts."

**The routing split**

The ambient monitor's decision call uses llama3.2:3b — it's a ~200-token JSON response and runs in ~0.8s warm. The actual content generation (notes from PDF, quiz questions) routes to a larger model: `AGENT_MIN_TIERS["content_generation"] = ModelTier.FRONTIER` — typically a 7B+ model depending on what you have loaded in Ollama.

**Benchmarks (M2 Pro, 16GB)**

| Task | llama3.2:3b | llama3.1:8b |
|------|-------------|-------------|
| Ambient decision (warm) | ~0.8s | ~1.4s |
| Notes from 10-page PDF | ~22s | ~8s |
| Quiz, 5 questions | ~18s | ~6s |
| VRAM loaded warm | ~2.3 GB | ~5.1 GB |

The split is intentional: lightweight model stays always-in-memory for fast decisions; the generation model loads on demand. The user-approval gate means the load overhead happens after they've confirmed, not speculatively.

MIT licensed. The ambient monitor is `apps/api/services/agent/ambient_monitor.py`. The approval flow lives in `apps/web/src/components/blocks/block-wrapper.tsx` (290 lines).

## c) r/Anki

**Title:** `I built a review-due counter that's filtered by the knowledge graph, not just time — here's how it differs from Anki's due count`

---

I'm building an open-source spaced repetition system on top of FSRS 4.5 (`apps/api/services/spaced_repetition/fsrs.py`, 304 lines) and recently shipped a Daily Digest block that surfaces four stats in a 2×2 grid:

1. Average mastery (% of concepts above retrieval threshold)
2. Concepts mastered (N / total)
3. **Review due count** (from the LECTOR/FSRS queue)
4. Total study time

The review-due count is not a simple "cards past due date" number — it's the output of LECTOR (`apps/api/services/lector.py`, 344 lines), a knowledge-graph-aware scheduling layer on top of FSRS.

**How it differs from Anki's due count**

Anki's due count is time-based: cards past their scheduled review date, sorted by how overdue they are. Our due count runs LECTOR prioritization first:

1. **Prerequisite ordering**: if you have a card for "gradient descent" and one for "backpropagation", the prerequisite surfaces first regardless of which is more overdue by FSRS.
2. **Confusion pair co-review**: concepts historically confused with each other (identified from wrong-answer analysis) are grouped into the same session.
3. **Cognitive load gate**: when current cognitive load score ≥ 0.7, new card introductions are suppressed — the count shows only existing due cards.

The number in the digest reflects the LECTOR-filtered queue, not the raw FSRS due list. It's typically smaller than the raw count — and the implied session is semantically ordered. You're not just reviewing what's overdue; you're reviewing in an order that respects what you know and what you've confused.

**One design difference worth noting**

Anki shows the due count on the deck list before you start. Ours lives inside the daily digest block in the workspace, alongside mastery percentage. The intent: the count isn't a score to optimize, it's context for the session you're about to start. Seeing "23 due" next to "68% mastery" frames the review differently than a standalone number.

The Daily Digest block is 118 lines (`apps/web/src/components/blocks/blocks/summary-block.tsx`). It fetches mastery from the progress endpoint and the due count from the review session endpoint in parallel. MIT licensed.

## d) r/GetStudying / r/medicalschool

**Title:** `Switching from a task list to a timeline view was the thing that finally made my exam prep feel managed`

---

I want to share something that changed how I approach exam prep, because I don't see it discussed often: the difference between a **task list** and a **timeline view** for study planning.

I've been scheduling study sessions in to-do lists for years. Read chapter 5. Complete practice set 3. Review unit 2 flashcards. Tasks checked off, progress made — and every exam I'd reach the final week and discover 4 chapters I'd mentally filed under "handle later."

The problem isn't discipline. It's representation. A task list tells you **what** to do. It doesn't tell you how much time you have, whether you're on track, or where the cliff edges are — the weeks where you have 3 exams and a lab report due simultaneously.

**What a timeline view shows that a list doesn't**

When I started plotting my study tasks against actual calendar weeks — exam dates as anchors, one row per subject — three things became immediately visible:

1. **Dead time**: weeks with nothing scheduled appear as obvious blank gaps, not invisible absences in a list
2. **Compression zones**: three weeks before finals, I had six topics requiring 4+ hours each stacked into the same window
3. **Buffer distribution**: I had 6 weeks before the exam, but content was concentrated in weeks 1–2 and 5–6; weeks 3–4 were empty without my realizing it

None of this was visible from the task list. The tasks were correct. The time shape was invisible.

**How to build this habit without special tools**

Take your task list. For each item, estimate required time. Plot it on a physical calendar or a simple spreadsheet — one column per week, exam dates marked in red. Color-code by subject. Step back and look at the *shape* of the plan, not just its contents.

The shape is what your brain processes quickly: am I front-loaded or back-loaded? Where's the dead zone? Where's the crunch?

I've also been using an open-source study tool I'm building (OpenTutor) that recently added a timeline tab to the study plan section, which does this automatically from your uploaded material and deadlines. But the manual spreadsheet version works just as well and takes ten minutes to set up.

The underlying insight: study planning is a time-budget problem as much as a content problem. You can have a perfect task list and still run out of time because the distribution is wrong. Switching to timeline thinking was the thing that fixed that for me.

What does your pre-exam planning look like? Do you think in tasks or in calendar weeks?

## e) Dev.to / Hashnode — Technical Article

**Title:** How I built a progressive feature unlock system for my AI learning app

---

OpenTutor is a local-first AI learning platform: upload a PDF, get notes, flashcards, quizzes, and an adaptive tutor running entirely on your machine with Ollama. The workspace is 12 composable blocks — notes, quiz, flashcards, knowledge graph, review queue, forecast, weak spots, daily digest, study plan, and more.

The first version dumped all 12 blocks in front of every new user. This was a mistake. A student who just uploaded their first document doesn't have enough content for a knowledge graph to be meaningful. A student who has never gotten a wrong answer has no data for a "Weak Spots" block to surface. Showing all features upfront overwhelms new users and undersells blocks that depend on accumulated data to do anything useful.

The usual solution for commercial products is paywalls. For an open-source local app, that's not a relevant constraint. I tried time-gating — unlock after N days — but calendar time doesn't proxy readiness. A student might use the app intensively for two days or casually over two months. The same wall hits both of them in the wrong place.

The solution: **behavioral unlocks**. Features unlock when the student demonstrates the behavior that makes them meaningful.

### The unlock logic

The core of the system is a single pure function in `apps/web/src/lib/block-system/feature-unlock.ts`:

```typescript
export interface UnlockContext {
  sourceDocCount: number;   // documents uploaded to this course
  practiceAttempts: number; // total quiz/flashcard attempts
  hasWrongAnswer: boolean;  // ever gotten one wrong
  sessionCount: number;     // course visits
  totalCourses: number;
  mode?: LearningMode;
  hasDeadline: boolean;
}

export function isBlockUnlocked(
  type: BlockType,
  ctx: UnlockContext,
): { unlocked: boolean; reason?: string; unlockHint?: string } {
  if (ALWAYS_AVAILABLE.includes(type)) {
    return { unlocked: true };
  }

  switch (type) {
    case "knowledge_graph":
      return ctx.sourceDocCount >= 3
        ? { unlocked: true }
        : {
            unlocked: false,
            reason: "Unlocks after uploading 3+ source documents",
            unlockHint: `${ctx.sourceDocCount}/3 documents`,
          };

    case "forecast":
      return ctx.practiceAttempts >= 50
        ? { unlocked: true }
        : {
            unlocked: false,
            reason: "Unlocks after 50+ practice attempts",
            unlockHint: `${ctx.practiceAttempts}/50 attempts`,
          };

    case "wrong_answers":
      return ctx.hasWrongAnswer
        ? { unlocked: true }
        : {
            unlocked: false,
            reason: "Unlocks after your first incorrect answer",
            unlockHint: "Answer a question incorrectly to unlock",
          };

    case "agent_insight":
      return ctx.sessionCount >= 3
        ? { unlocked: true }
        : {
            unlocked: false,
            reason: "Unlocks after 3+ learning sessions",
            unlockHint: `${ctx.sessionCount}/3 sessions`,
          };

    default:
      return { unlocked: true };
  }
}
```

The conditions were chosen to match the minimum viable data each feature needs:

- **Knowledge Graph** needs multiple documents to build a non-trivial concept map. One 30-page PDF produces a flat graph with no interesting prerequisite structure.
- **Forecast** needs enough practice history for trajectory prediction to be meaningful. Below ~50 attempts the confidence interval is too wide to be useful.
- **Weak Spots** has nothing to show before a wrong answer exists — an empty block is confusing, not helpful.
- **Agent Insight** needs behavioral history before the AI can generate useful workspace suggestions. Three sessions is enough to observe patterns; fewer produces generic output.

The context is computed from `localStorage`, updated in real time as the student uses the app. No backend round-trip needed for unlock checks — the data already lives client-side.

### The locked overlay

In `apps/web/src/components/blocks/block-wrapper.tsx` (290 lines), the block wrapper checks unlock state on every render and renders a frosted-glass overlay when locked:

```tsx
// Feature unlock check — runs on every render from localStorage state
const { unlocked, unlockHint, reason: unlockReason } = useMemo(() => {
  const ctx = {
    ...getUnlockContext(courseId, courses.length),
    mode: currentMode ?? undefined,
  };
  return isBlockUnlocked(block.type, ctx);
}, [block.type, courseId, courses.length, currentMode]);

// ...later in JSX, inside the block content area:
{!unlocked && (
  <div className="absolute inset-0 flex flex-col items-center justify-center gap-3 bg-background/80 backdrop-blur-sm rounded-b-2xl z-20 group">
    <div className="flex flex-col items-center gap-2 p-4 text-center">
      <div className="flex size-10 items-center justify-center rounded-full bg-muted ring-2 ring-border group-hover:ring-brand/30 transition-all">
        <Lock className="size-5 text-muted-foreground group-hover:text-brand transition-colors" />
      </div>
      <p className="text-sm font-medium text-foreground">{entry.label} 已锁定</p>
      {unlockHint && (
        <p className="text-xs text-muted-foreground max-w-[200px]">{unlockHint}</p>
      )}
      {unlockReason && (
        <p className="text-[11px] text-muted-foreground/70 max-w-[220px]">{unlockReason}</p>
      )}
    </div>
  </div>
)}
```

Three deliberate choices:

1. **The component renders underneath the overlay**, not hidden. Students see a blurred preview of what they're unlocking — a graph with nodes, an axis, a card list. Opacity communicates value before access.
2. **The overlay shows a progress counter** via `unlockHint` — "2/3 documents", "47/50 attempts". This gives a concrete next action, not a vague "unlock later" message.
3. **The lock icon animates on hover** (`group-hover:ring-brand/30`, `group-hover:text-brand`). Small, but it makes the overlay feel interactive rather than a dead wall.

### The block registry

Each block registers its metadata in `apps/web/src/lib/block-system/registry.ts`, where all 12 block types are defined. Blocks load their components lazily via `React.lazy()`:

```typescript
export const BLOCK_REGISTRY: Record<BlockType, BlockRegistryEntry> = {
  knowledge_graph: entry(
    "knowledge_graph", "Knowledge Graph", "知识图谱", "GitBranch",
    "LOOM concept relationship map",
    "medium", {},
    () => import("@/components/blocks/blocks/knowledge-graph-block"),
  ),
  summary: entry(
    "summary", "Daily Digest", "学习摘要", "Newspaper",
    "Today's learning snapshot: mastery, quiz accuracy, review due",
    "small", {},
    () => import("@/components/blocks/blocks/summary-block"),
  ),
  // ... 10 more
};
```

The registry is the single source of truth for labels, icons, default sizes, and lazy loaders. Locked blocks still appear in the workspace (header visible with correct label and icon), just with their content behind the overlay. The unlock check lives in the wrapper, not the registry, which keeps the registry as pure configuration.

### What changed after shipping this

New users now see 5 blocks by default: notes, quiz, flashcards, chapters, review — all always-available, all immediately useful on a fresh course. As they use the app, additional blocks surface. The first wrong answer triggers Weak Spots. After the third document, Knowledge Graph appears. After 50 practice attempts, Forecast shows up.

The feedback from early users was that the progressive unlocks made the app feel like it was "learning what you need" rather than "hiding features." Whether the thresholds are exactly right is an empirical question — 50 attempts for Forecast might be too high. We're watching session recordings to calibrate.
