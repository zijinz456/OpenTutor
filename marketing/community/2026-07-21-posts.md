## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – AI tutor that routes model tiers by heuristic, not a meta-LLM`

---

There's a pattern in AI app development I find annoying: to decide which model to call, you call a model. A small LLM classifies the request, then you send it to the appropriate LLM. You've added latency and cost to save latency and cost.

OpenTutor is a local-first AI learning platform (FastAPI + Next.js 16, runs on Ollama). Every message the tutor receives needs to be routed to one of three model tiers — fast (short greeting, preference change), standard (teach a concept, generate a quiz), or frontier (multi-step study plan, code execution). I wanted routing logic that costs zero tokens.

The solution is a pure scoring function in `apps/api/services/llm/complexity.py` (196 lines). It takes the message, classified intent, active scene, conversation depth, and a RAG flag. It assigns points across five dimensions: message length (0–150), intent (greeting = 30, assess = 200, curriculum = 200, plan = 250), scene (exam_prep = 80, review_drill = 40), conversation depth (capped at 8 extra messages × 10 pts), and complexity-marker phrases (things like "step by step" = 40, "prove" = 50, "compare" = 30). The score ranges 0–780. Below 150 → fast tier; 150–499 → standard; 500+ → frontier. The function returns in microseconds, no model call.

There's a per-agent minimum tier floor: `planning` and `code_execution` agents always go frontier regardless of score. `teaching` and `exercise` agents floor at standard. This catches cases where the intent classification wins over the message alone — a two-word message saying "make me a plan" scores low on length but `plan` intent routes it to frontier.

The circuit breaker in `apps/api/services/llm/circuit_breaker.py` (83 lines) handles what happens when the frontier model is unavailable: progressive cooldown steps [5, 10, 20, 60]s, opens after 3 consecutive failures, auto-resets at 120s. Background health probes every 30s update a provider cache so the router can skip unhealthy clients without waiting for a timeout. 372 pytest functions, MIT, `docker compose up -d --build`.

GitHub: https://github.com/zijinz456/OpenTutor

## b) r/LocalLLaMA

**Title:** `I use llama3.2:3b for fast routing decisions and a circuit breaker to auto-fallback when bigger models fail — here's the setup`

---

OpenTutor is an open-source AI learning platform that defaults to Ollama + llama3.2:3b. I want to share two patterns from the LLM router (`apps/api/services/llm/router.py`, 399 lines) that took a few iterations to get right.

**Problem 1: routing overhead**

The obvious approach to "use a small model for simple questions and a bigger model for hard ones" is to call the small model first to classify. The overhead is ~0.8s on llama3.2:3b warm — fine for one request, but compounding. The replacement is a heuristic scoring function with no LLM call: message length, classified intent, scene context, conversation depth, and complexity keywords. Scores route to three tiers — fast/standard/frontier. On my M2 Pro it costs ~0.3ms. The function covers education-domain patterns well enough that I haven't needed a meta-LLM.

**Problem 2: cascade failures when Ollama hiccups**

Local LLMs go unhealthy in ways cloud APIs don't: process restarts, VRAM contention, model swaps mid-session. Without a circuit breaker you get requests stacking on an unhealthy provider, timeouts, and a jammed app.

The circuit breaker mixin tracks failures per provider with progressive cooldown: first failure = 5s pause, second = 10s, third = 20s, fourth and beyond = 60s. After 3 consecutive failures the circuit opens entirely — no further requests — for 120s, then auto-resets. A background probe loop pings all registered providers every 30s and caches their health state, so the router gets a warm health signal without blocking the request path.

```python
COOLDOWN_STEPS = [5, 10, 20, 60]     # seconds, progressive
CIRCUIT_OPEN_THRESHOLD = 3           # failures before open
CIRCUIT_RESET_TIMEOUT = 120          # seconds before auto-reset
```

The fallback chain: model variant → primary → fallback order. If you have llama3.2:3b loaded and llama3.1:8b registered as fallback, a failed 8b response auto-routes to 3b on the next attempt. Not ideal (quality drop), but it's better than a frozen app.

Multi-provider setup example:

```bash
LLM_PROVIDER=ollama
LLM_MODEL=llama3.2:3b
LLM_FALLBACK_PROVIDER=deepseek
DEEPSEEK_API_KEY=sk-...
```

MIT licensed. 63 test files, 372 test functions. Runs with `docker compose up -d --build`.

## c) r/Anki

**Title:** `I upgraded from FSRS-4.5 to FSRS-5/6 in my open-source SRS — here's what actually changed in the formulas`

---

I've been building OpenTutor, an open-source learning platform, on top of FSRS. When I started, I implemented FSRS-4.5 (17 parameters). I've since upgraded to FSRS-5/6 (21 parameters). I want to document what actually changed in the math, because the version names don't make it obvious.

**What FSRS-5/6 adds over 4.5**

Three specific changes to the formulas:

*1. Exponential difficulty initialization (was linear)*

FSRS-4.5: `D₀ = w₄ - (G - 3) × w₅`

FSRS-5/6: `D₀ = w₄ - exp(w₅ × (G - 1)) + 1`

The exponential form spreads initial difficulty values further apart across the rating scale. Cards rated "Easy" start at a lower difficulty than they would under the linear formula; cards rated "Again" start higher. This gives the scheduler more room to differentiate from the first review.

*2. Same-day review stability formula (new in FSRS-5)*

FSRS-4.5 didn't have a separate path for same-day reviews — if you reviewed a card twice in a session, the second review ran through the standard stability update. FSRS-5 adds:

`S′ = S × exp(w₁₇ × (G − 3 + w₁₈)) × S^(−w₁₉)`

This prevents stability inflation from repeated intra-day reviews (a pattern Anki users know from hammering a card in a single sitting).

*3. Trainable forgetting curve decay (w₂₀)*

FSRS-4.5 forgetting curve: `R(t) = (1 + t/9S)^(-1)`

FSRS-5/6: `R(t) = (1 + factor × t/S)^(-decay)` where `decay = w₂₀`, `factor = 9 × decay`

When `decay = 1.0`, this reduces exactly to the FSRS-4.5 formula — backward compatible. When trained, `w₂₀` adjusts the shape of the forgetting curve to fit the user's actual retention data. The default is `1.0`; the parameter only becomes meaningful once you have enough review history to optimize against.

**vs. Anki's SM-2**

SM-2 uses a fixed forgetting curve and interval multiplier that doesn't change per card. FSRS models stability as a per-card parameter that grows with each successful review and shrinks on lapse. The FSRS-5/6 paper claims 30%+ improvement in retention prediction over SM-2 on real Anki review datasets.

Our implementation is in `apps/api/services/spaced_repetition/fsrs.py` (304 lines) and adds a LECTOR scheduling layer on top for concept-graph-aware review ordering. MIT licensed.

## d) r/GetStudying / r/medicalschool

**Title:** `I started tracking five things after every study session and it changed how I plan the next one`

---

I want to share a self-monitoring habit that made a real difference for me during exam prep, because I don't see it discussed much: tracking your own cognitive signals, not just your content progress.

The standard study log is output-focused — chapters read, questions completed, hours logged. That tells you *what* you did. It doesn't tell you *how well your brain was working* while you did it.

Here's what I started tracking instead:

**1. Error rate in the last 30 minutes**

Not total accuracy — just the last 30 minutes. If I'm getting 5+ questions wrong in a tight window despite knowing the material cold from last session, that's a signal my processing capacity is degraded, not that I've forgotten the content.

**2. Message length when asking my AI tutor questions**

This sounds weird, but I noticed a pattern: when I'm focused, I write detailed questions. When I'm fried, my questions get terse — "what is this again?" instead of "I don't understand why the derivative of sin(x) is cos(x) and not -cos(x)." The brevity of how I ask for help is a reliable signal of how much bandwidth I have left.

**3. How often I typed "help" or "explain" unprompted**

One or two genuine confusion questions in a session = normal learning. Constantly reaching for help = I'm past the point where new content sticks.

**4. Session length since last break**

After about 45 minutes of focused work, most people's recall drops noticeably. I started treating 45 minutes as a hard ceiling and resetting deliberately.

**5. How I feel about starting the next topic**

Not enthusiasm — just "do I actively dread this?" Active dread before starting a new topic is a strong signal to stop, not push through. Material absorbed while dreading the process doesn't consolidate as well.

Tracking these five things takes about two minutes after each session block. The pattern I found: sessions where signals 1–3 were high and I kept going anyway produced almost no durable learning. I was technically studying; my brain wasn't encoding.

The action: when two or more of these signals are high, I stop, do something physical for 10–15 minutes, and start fresh — even if my schedule says I have another hour to go. The hour I take at 80% cognitive load is worth less than 30 focused minutes at 100%.

(For what it's worth, the study tool I've been building — OpenTutor — does try to track these signals automatically and adjusts difficulty when it detects overload. But the manual version of this habit works too. The valuable thing is just being aware of the signals.)

## e) Dev.to / Hashnode — Technical Article

**Title:** How I built a heuristic model router for my AI tutor — no meta-LLM required

---

Every AI application that routes between model tiers eventually hits the same question: how do you decide which tier to use? The common answer is "ask a small model to classify the request, then route." I tried that. The overhead compounds, and you've added a dependency on a second model to manage a first model.

OpenTutor is a local-first AI learning platform — FastAPI backend, Next.js 16 frontend, runs with Ollama. Every message goes through a tutor agent that may need to teach a concept, generate a quiz, build a study plan, or respond to a one-word acknowledgment. These have wildly different complexity requirements. I needed routing that costs zero tokens.

The result is a pure scoring function in `apps/api/services/llm/complexity.py` (196 lines). Here's how it works.

### Three tiers

```python
class ModelTier(str, Enum):
    FAST = "fast"        # greetings, simple preference changes
    STANDARD = "standard" # teaching, exercises, review — most requests
    FRONTIER = "frontier" # planning, code execution, multi-step reasoning
```

Thresholds are numeric: score < 150 → FAST, 150–499 → STANDARD, ≥ 500 → FRONTIER. The score range tops out at ~780.

### The scoring function

```python
def score_complexity(
    message: str,
    intent: str = "general",
    scene: str = "study_session",
    history_length: int = 0,
    has_rag_context: bool = False,
) -> int:
    score = 0
    score += _score_message_length(message)    # 0–150
    score += _score_intent(intent)             # 0–250
    score += _score_scene(scene)              # 0–80
    score += _score_conversation_depth(history_length)  # 0–80
    score += _score_complexity_markers(message)          # 0–100 (capped)
    if has_rag_context:
        score += 50
    return score
```

Five independent dimensions, each with its own scoring logic.

**Intent scoring** is the heaviest dimension because intent tells you more about what the model needs to do than message length does:

```python
intent_scores = {
    "preference": 30,   # "change font size" — no reasoning needed
    "general": 50,
    "layout": 30,
    "scene_switch": 30,
    "learn": 150,       # concept explanation — needs synthesis
    "quiz": 180,        # question generation — needs content understanding
    "review": 160,
    "assess": 200,
    "curriculum": 200,
    "plan": 250,        # multi-step planning — frontier territory
    "code": 250,
}
```

Intent is classified separately (a simple keyword/regex classifier, also zero-LLM). A two-word message like "make a plan" has a very low length score but a plan intent score of 250, which routes it correctly to frontier.

**Complexity markers** catch reasoning-heavy requests that don't map to a specific intent:

```python
_COMPLEXITY_PATTERNS = [
    (re.compile(r"\bstep[- ]by[- ]step\b", re.IGNORECASE), 40),
    (re.compile(r"\bprove\b", re.IGNORECASE), 50),
    (re.compile(r"\bderive\b", re.IGNORECASE), 40),
    (re.compile(r"\banalyze\b|\banalyse\b", re.IGNORECASE), 30),
    (re.compile(r"\bcompare\b", re.IGNORECASE), 30),
    # CJK equivalents — the app is bilingual
    (re.compile(r"证明|推导"), 50),
    (re.compile(r"比较|对比"), 30),
]
```

The score is capped at 100 regardless of how many patterns match — a request doesn't become twice as frontier just because it says "compare and prove."

### Per-agent minimum tier floors

The score drives the initial tier, but each agent has a minimum it won't go below:

```python
AGENT_MIN_TIERS: dict[str, ModelTier] = {
    "preference": ModelTier.FAST,
    "teaching": ModelTier.STANDARD,
    "exercise": ModelTier.STANDARD,
    "assessment": ModelTier.STANDARD,
    "planning": ModelTier.FRONTIER,
    "code_execution": ModelTier.FRONTIER,
}

def resolve_tier(agent_name: str, message: str, ...) -> ModelTier:
    scored_tier = _tier_from_score(score_complexity(message, ...))
    agent_min = AGENT_MIN_TIERS.get(agent_name, ModelTier.STANDARD)
    tier_order = {ModelTier.FAST: 0, ModelTier.STANDARD: 1, ModelTier.FRONTIER: 2}
    return scored_tier if tier_order[scored_tier] >= tier_order[agent_min] else agent_min
```

The `planning` agent always gets frontier regardless of what it says. This protects against the edge case where a brief planning message gets routed to standard and produces a shallow study plan.

### Circuit breaker for when the frontier model is down

The routing is useless if it routes to an unhealthy provider. The circuit breaker in `apps/api/services/llm/circuit_breaker.py` (83 lines) tracks health per provider:

```python
COOLDOWN_STEPS = [5, 10, 20, 60]  # progressive cooldown in seconds
CIRCUIT_OPEN_THRESHOLD = 3         # failures before circuit opens
CIRCUIT_RESET_TIMEOUT = 120        # seconds before auto-reset
```

On each failure, the cooldown step index advances. After 3 consecutive failures the circuit opens — no requests pass until 120s have elapsed. A background health probe loop runs every 30s and pings all registered providers, updating a cache. The router consults this cache instead of blocking on a live probe.

The fallback chain (`model_variants → primary → fallback_order`) means a failed frontier model can fall back to standard-tier while the circuit is open. Not ideal — quality drops — but the app stays responsive.

### What the function doesn't do

It doesn't look at the content of the user's material (PDFs, notes). The complexity score is request-level, not content-level. A short question about a complex topic scores low on length — but intent + markers usually compensate. In cases where they don't, the agent minimum tier acts as a floor.

It also doesn't train. The thresholds (150/500) and intent scores were set by hand based on observed routing errors and adjusted until misrouting was rare. A data-driven approach would collect user-rating feedback and fit them. That's on the roadmap; the heuristic is good enough to ship.
