## a) Hacker News — Show HN

**Title:** `Show HN: OpenTutor – a cognitive load auto-tuner that stops signals from crying wolf`

---

Most adaptive tutoring systems treat cognitive load as a fixed function of behavior: long pause → overloaded, short message → frustrated, wrong answer streak → struggling. The problem is that students have baselines. Some people always write one-sentence messages. Some students always hit 3+ wrong answers before they click. If you weight those signals uniformly you get score saturation — the system thinks the student is always overloaded and adapts down to "explain like I'm five" permanently.

I shipped a fix for this in OpenTutor (a local-first AI tutoring platform built on FastAPI + Next.js) and wanted to share the design because it's simple in a way I don't see discussed. The file is `apps/api/services/cognitive_load_tuning.py` (136 lines). It maintains a per-user sliding window of 20 recent signal values and applies a small weight multiplier adjustment after each computation:

```python
WINDOW_SIZE = 20
HIGH_SIGNAL_THRESHOLD = 0.65   # Mean above this → "persistently high"
WEIGHT_DECAY_FACTOR   = 0.90   # Multiply weight by this when high
MIN_WEIGHT_RATIO      = 0.40   # Floor — never suppress entirely
RECOVERY_FACTOR       = 1.04   # Slowly recover toward 1.0 when signal drops
```

If a signal's 20-message rolling mean stays above 0.65, the weight for that signal is nudged down by 10% (minimum 40% of original). When it drops below the threshold, it nudges back up by 4% per turn. The result: a student who always writes short messages gradually reduces the brevity signal's influence on their personal score, while a student who only occasionally writes short messages (sudden behavior change) gets full signal weight on that event.

No ML. No separate training step. No per-user model file. The state lives in a bounded in-memory dict (max 200 users) and is reset on a long session gap. There's also an LRU-style eviction when the cache is full — evict the least active user by window fullness.

The broader system it feeds into is 12 behavioral signals (error streaks, pause duration, NLP affect analysis, help-seeking keywords, answer hesitation timing, etc.) that combine into a 0.0–1.0 score injected into the tutor's system prompt each turn. 996 test functions, MIT, `docker compose up -d --build`.

## b) r/LocalLLaMA

**Title:** `I added LLM-graded coding questions to a local study app — here's how it's scored without a rubric`

---

OpenTutor is an open-source AI learning platform I'm building. We recently shipped coding questions as a 7th question type (alongside MCQ, short answer, fill-in-blank, true/false, matching, ordering). I want to share the grading approach because it's one of the more interesting local-LLM use cases I've built.

**The problem with coding question grading**

You can't just check exact output. A student's solution to "implement binary search" might be correct but structured differently from the sample answer. Pure string matching fails. You need something that understands code semantics, edge case handling, and whether the logic is sound — which points to LLM grading.

**What we do**

The coding question type sends the student's code and the question to the local model for evaluation. The grader prompt doesn't use a fixed rubric — it asks the model to assess:

1. **Correctness**: Does the logic produce the right output for the given problem?
2. **Completeness**: Does it handle the described cases (including any mentioned edge cases)?
3. **Clarity**: Is the code readable without being unnecessarily clever?

The model returns a structured JSON verdict with a score (0–100), a pass/fail flag at threshold 60, and a 2-3 sentence explanation the student sees immediately.

**Why local models handle this surprisingly well**

For most coding questions in an educational context — implementing sorting algorithms, writing a recursive function, basic data manipulation — llama3.2:3b is actually good enough as a grader. The failure mode is subtle correctness errors in long complex programs, which we avoid by constraining question scope. For multi-step reasoning or when the question involves complex algorithms, the complexity scorer routes to a larger frontier model.

The frontier tier matters here: coding grading routes to `AGENT_MIN_TIERS["code_execution"] = ModelTier.FRONTIER` regardless of message complexity score. A short 10-line submission still needs the bigger model.

**RAM and response time**

- llama3.2:3b loaded warm: ~2.3 GB VRAM, ~0.8s/response for standard tutoring
- Coding grading on frontier (e.g. 7B model): ~4.2 GB VRAM, ~2.5s response warm, ~9s cold start
- The grading call is async and doesn't block the UI — student sees a spinner then gets the verdict

MIT licensed, runs with `docker compose up -d --build`. The complexity router is `apps/api/services/llm/complexity.py`.

## c) r/Anki

**Title:** `I added a sliding window weight dampener to my FSRS implementation — sharing the approach`

---

I'm building an open-source spaced repetition system on top of FSRS-5/6 (21 parameters, upgraded from FSRS-4.5's 17) and ran into a problem I haven't seen discussed in the context of algorithmic review scheduling: **baseline normalization for behavioral signals**.

Here's the context: on top of FSRS card-level scheduling, OpenTutor has a 12-signal cognitive load detector that adjusts tutor behavior in real time. These signals include wrong-answer streaks, message brevity, answer hesitation (median of last 10 quiz response times), NLP affect analysis, and 8 others. The cognitive load score influences which cards get promoted in the review queue — specifically, if the score is high, we hold off on introducing new cards and focus on consolidating existing ones.

The problem: some users have high-baseline signals that aren't informative. A student who types brief messages by habit keeps triggering the brevity signal even when they're not cognitively overloaded. The weighted combination saturates, the tutor over-simplifies, and review prioritization gets miscalibrated.

**The fix: per-signal sliding window dampening**

```python
# cognitive_load_tuning.py (136 lines)
WINDOW_SIZE = 20
HIGH_SIGNAL_THRESHOLD = 0.65
WEIGHT_DECAY_FACTOR = 0.90   # -10% per turn when persistently elevated
MIN_WEIGHT_RATIO = 0.40      # never suppress below 40%
RECOVERY_FACTOR = 1.04       # +4% per turn when signal drops
```

After each cognitive load computation, we record all 12 raw signal values into per-user rolling windows (deque of maxlen 20). If a signal's rolling mean stays above 0.65, its weight contribution is reduced by 10% (floored at 40%). When it drops back below the threshold, it recovers by 4% per turn. This means an always-brief typist eventually has their brevity signal dampened to ~40% of normal weight, while someone who suddenly starts writing short messages (a real behavioral change) gets full signal weight on that event.

**Connection to FSRS**

The cognitive load score feeds back into the LECTOR review queue as a "suppress new cards" signal. When score ≥ 0.7 for two consecutive turns, the review session only surfaces due cards — no new introductions. When it drops below 0.4, new card introduction resumes. This is separate from the FSRS scheduling itself; FSRS still computes retrievability normally. The cognitive load layer just gates what the student *sees* from the due queue.

For reference, the FSRS scheduler is `apps/api/services/spaced_repetition/fsrs.py` (304 lines) and the LECTOR knowledge-graph-aware layer is `apps/api/services/lector.py` (344 lines). 996 test functions total. MIT licensed.

## d) r/GetStudying / r/medicalschool

**Title:** `The thing that helped my weekly review sessions actually stick: a daily digest instead of re-reading notes`

---

I want to share a study habit that changed how I approach end-of-week review, because I stumbled into it accidentally and I don't see it recommended often.

The context: I was studying for a dense subject (molecular biology) with a lot of interconnected concepts. Every Sunday I'd sit down to review the week and feel completely lost — not because I hadn't studied, but because the week's content had blurred together. I'd re-read my notes, feel like I understood them, and then fail the practice test anyway.

The problem wasn't the review session. It was that I had no anchor. I knew I'd covered a lot, but I couldn't quickly reconstruct *what* I'd actually covered and which parts had been shaky.

**What changed: a five-minute daily digest habit**

At the end of each study session, before closing my notes, I write three bullet points:

1. The one concept I feel least confident about from today
2. The one concept I feel most confident about
3. One question I'd ask myself in a week to test whether I retained today's work

That's it. Takes five minutes. The key is the *third* one — writing the future test question while the material is fresh encodes what actually matters, not just what I covered.

When Sunday comes around, I have a list of actual recall questions I wrote myself, organized by which concepts I flagged as shaky. I do the review session against those, not against my full notes. It's dramatically faster and hits the right gaps instead of re-covering the confident material.

**Why this works better than re-reading**

Re-reading creates familiarity — the notes look familiar, which feels like understanding. Self-quizzing from questions you wrote yourself creates retrieval — you're actively reconstructing, not recognizing. The brain consolidates differently.

The flag-your-weakest-concept step forces you to be honest during the session, when you're still close enough to the material to know what actually confused you. By Sunday you've forgotten which parts were hard, so you review everything equally.

I've also been using an open-source study tool I'm building (OpenTutor) that recently added a daily summary block — it auto-generates this kind of digest from your session activity, with a section on shaky concepts from quiz wrong answers. But the manual habit works just as well if you're already set up with your own note system.

## e) Dev.to / Hashnode — Technical Article

**Title:** How I built a self-calibrating cognitive load sensor that stops crying wolf

---

Adaptive AI tutoring systems have a calibration problem no one talks about: the same behavior means different things for different students. A one-word message might mean frustration or it might just mean someone who types tersely. Three wrong answers in a row might mean genuine confusion or a hard topic that normally takes a few attempts. If you build a fixed-weight signal aggregator and treat all students the same, you get a system that's miscalibrated for anyone who deviates from the assumed baseline.

In OpenTutor — a local-first AI learning platform built on FastAPI + Next.js — I recently shipped a fix for this: a per-user sliding window weight dampener that adjusts signal influence based on each student's observed behavior. Here's how it works and why I designed it this way.

### The problem: signal saturation

OpenTutor's cognitive load detector (`apps/api/services/cognitive_load.py`, 488 lines) aggregates 12 behavioral signals into a 0.0–1.0 score that's injected into the tutor's system prompt each turn. The signals include message brevity, wrong-answer streaks, answer hesitation timing (median of last 10 quiz response times), NLP affect analysis, help-seeking keyword detection, pause duration between messages, and 6 others.

Each signal contributes a weighted fraction to the total. The score drives two things: the tutor's explanation depth/vocabulary, and the workspace block layout (high load removes non-essential blocks; low load restores them).

The problem I ran into: students with persistently high-baseline signals cause the score to saturate near 1.0, so the tutor is always in "simplify everything" mode even when the student is fine. A student who habitually types short messages keeps triggering the brevity signal. A student who's genuinely slow at typing triggers the hesitation signal constantly. The signal is real but not *informative* — it reflects baseline behavior, not a change in state.

### The solution: per-signal sliding window dampening

The fix is in `apps/api/services/cognitive_load_tuning.py` (136 lines). After every cognitive load computation, we call `update_and_get_multipliers()` with the raw signal values. It maintains a per-user rolling window of 20 recent values for each signal and adjusts a weight multiplier accordingly:

```python
WINDOW_SIZE = 20
HIGH_SIGNAL_THRESHOLD = 0.65   # Mean above this → "persistently high"
WEIGHT_DECAY_FACTOR = 0.90     # Reduce weight by 10% when persistently elevated
MIN_WEIGHT_RATIO = 0.40        # Never suppress below 40% of original weight
RECOVERY_FACTOR = 1.04         # Nudge weight back up when signal drops
MAX_WEIGHT_RATIO = 1.0         # Cap at original weight
```

```python
def update_and_get_multipliers(
    user_id: str,
    signals: dict[str, float],
) -> dict[str, float]:
    state = _get_or_create(str(user_id))

    for signal_name, value in signals.items():
        if signal_name not in state.windows:
            state.windows[signal_name] = collections.deque(maxlen=WINDOW_SIZE)
            state.multipliers[signal_name] = 1.0

        state.windows[signal_name].append(float(value))
        dq = state.windows[signal_name]
        if len(dq) < 5:
            continue   # Need at least 5 data points before adjusting

        mean = sum(dq) / len(dq)
        current = state.multipliers[signal_name]

        if mean >= HIGH_SIGNAL_THRESHOLD:
            new = max(MIN_WEIGHT_RATIO, current * WEIGHT_DECAY_FACTOR)
            state.multipliers[signal_name] = new
        else:
            new = min(MAX_WEIGHT_RATIO, current * RECOVERY_FACTOR)
            state.multipliers[signal_name] = new

    return dict(state.multipliers)
```

The caller applies these multipliers before the next `compute_cognitive_load()` call:

```python
load_result = await compute_cognitive_load(...)
multipliers = update_and_get_multipliers(str(user_id), load_result["signals"])
# Next call uses: effective_weight = settings.cognitive_load_weight_X * multipliers.get("X", 1.0)
```

### What this does in practice

For a student who always types brief messages: after 20 turns, the brevity signal's rolling mean is consistently above 0.65. The multiplier decays to ~0.54 (0.90^6 ≈ 0.53 after 6 consecutive high-mean windows). This student's brevity signal now contributes ~54% of its original weight — still informative, but not dominating. If they suddenly type a very long message (genuine change in behavior) followed by short ones again, the mean drops slightly, recovery begins, and the signal regains influence slowly.

For a student with normal behavior: signals fluctuate naturally. No signal's rolling mean stays consistently above 0.65. Multipliers stay near 1.0. The system behaves as designed without adjustment.

### The memory footprint

State lives in a bounded in-memory dict capped at 200 users (`_MAX_CACHE_SIZE`). When the cache is full, we evict the least active user by rolling window fullness — the user with the fewest accumulated signal observations:

```python
evict = min(
    _store,
    key=lambda k: sum(len(dq) for dq in _store[k].windows.values()),
)
```

No database write. No model file. The state resets on server restart. For the current single-user local-first beta, this is fine. For a multi-user deployment, you'd persist the multipliers alongside the user session record — the `multipliers` dict is JSON-serializable.

### Why not just normalize globally?

The alternative is global baseline normalization — compute z-scores relative to all users. The problem is that baselines shift over time (a student progresses and naturally writes longer messages), and in a local-first app with a single user, there's no population to normalize against. Per-user sliding window dampening handles both cases without requiring any external calibration data.

### Connecting it to review scheduling

The cognitive load score feeds into LECTOR (`apps/api/services/lector.py`, 344 lines), the knowledge-graph-aware spaced repetition layer. When the score is high (≥ 0.7) for two consecutive turns, the review queue suppresses new card introductions — only due cards from the existing deck are shown. When it drops back below 0.4, new introductions resume. The FSRS scheduler (`apps/api/services/spaced_repetition/fsrs.py`, 304 lines) still computes retrievability normally; the cognitive load layer only gates what the student sees from the due queue.

This means the auto-tuner has a downstream effect on how aggressively the spaced repetition system introduces new material. A student who types tersely but is actually fine would previously get perpetually suppressed new-card introductions. With the dampener, their brevity signal calibrates to their baseline, and the review queue opens back up.
